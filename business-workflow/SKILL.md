# Business Workflow Design

本技能封装了 `workflow-viewer.html` + `data/*.js` 的完整工具链。
所有模板文件内置在技能的 `templates/` 目录下，无需依赖外部项目。

## 本技能规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | **结构化数据优先** | 业务工作流数据以 JS 文件为唯一权威源，`workflow-viewer.html` 只是渲染层 |
| 2 | **单一渲染器** | 所有领域共用 `workflow-viewer.html`，样式统一，维护成本低 |
| 3 | **域注册表同步** | 开工前先读取 `design/domain-registry.js`，改领域边界时必须先写注册表再写自己的数据（development-standard §5.3） |
| 4 | **决策可追溯** | 每次设计操作前，将原始输入原文追加到 `design/01-raw-input/{domain-slug}.md` |
| 5 | **CHANGELOG 必写** | 每次变更后在 `design/02-business-workflow/data/CHANGELOG.md` 记录 |
| 6 | **版本控制友好** | 数据文件为纯文本 JS，可 diff、可 review、可回滚 |
| 7 | **权限控制点同步设计** | 从 action 节点提取权限点——**business-workflow 是权限点的唯一定义源**，下游只做映射引用。详见 §3.4 |
| 8 | **角色同步定义** | 在权限点确定后同步定义角色及其权限聚合关系——**谁**通过**什么操作**完成**什么业务**。详见 §3.5 |

## 1. 设计流程总览

业务工作流设计遵循 **"5 步法"**流水线：

```
输入源                         处理步骤                         输出
┌────────────┐                ┌──────────────┐              ┌──────────────┐
│ 原始需求文档  │ ─────────────→ │ 1. 输入采集与  │              │ data/XX.js    │
│（readme/    │   需求文案       │    需求理解    │              │（JSON 数据源） │
│ 需求文件）   │                │              │              │ 包含：         │
├────────────┤                │ 2. 业务要素    │ ───────────→ │ - sections   │
│ 终端文字输入  │ ─────────────→ │    提炼分析    │              │   (文本块)    │
│（对话/分块   │   业务场景描述   │              │              │ - flowchart  │
│  迭代输入）  │                │ 3. 流程建模    │              │   (流程图)    │
├────────────┤                │   （含状态机）  │              │              │
│ 已有工作流   │ ─────────────→ │              │              │ workflow-    │
│ 数据文件     │   迭代基线      │ 4. 输出生成    │              │ viewer.html  │
└────────────┘                │              │              │（SVG 渲染器）  │
                              │ 5. 迭代修正    │              └──────────────┘
                              └──────────────┘
```

**每次被调用时，skill 必须先输出执行计划与用户确认，然后再开始执行。**

## 2. 输入规格

### 2.1 输入来源类型

| 来源类型 | 说明 | 使用方式 |
|---------|------|---------|
| **原始需求文档** | 项目中已存在的需求文档（如 `/design/01-raw-input/` 下的 Markdown 文件） | 读取全文，提取业务场景、角色、流程步骤、状态定义 |
| **终端文字输入** | 用户通过命令行/对话框直接输入描述 | 实时理解，按一段/一块处理，可多次迭代补充 |
| **分块输入** | 大量输入时分多次提供，每次处理一块 | 保持上下文，每块处理完后等待下一块 |
| **已有数据文件** | 已存在的 `data/XX-slug.js` 文件 | 作为迭代基线读取，增量修改 |

### 2.2 输入内容要求

| 输入内容 | 说明 | 必需程度 |
|---------|------|---------|
| 业务角色定义 | 有哪些角色参与（如用户、管理员、成员） | ★★★ 必需 |
| 核心流程步骤 | 业务从开始到结束的执行顺序 | ★★★ 必需 |
| 状态定义 | 业务对象有哪些状态（如 draft/active/suspended） | ★★★ 必需 |
| 状态转换条件 | 什么操作触发状态变更 | ★★☆ 推荐 |
| 业务规则约束 | 有哪些校验规则和限制条件 | ★★☆ 推荐 |
| 异常流程 | 失败/取消/回退的处理方式 | ★☆☆ 补充 |
| 设计决策说明 | 为什么这样设计的决策依据 | ★☆☆ 补充 |

## 3. 设计过程（5 步法）

### 步骤 1：输入采集与需求理解

**目标：** 全面理解业务领域的流程、角色、规则。

1. **识别输入源：**
   - **读取域注册表 `design/domain-registry.js`**（development-standard skill §5.3），获取当前领域 slug 列表。后续的 `{domain-slug}` 参数以此为准
   - 如果用户指定了需求文档路径 → 读取该文件
   - 如果用户通过终端描述 → 解析自然语言输入
   - 如果用户分块提供 → 每块到达时理解并整合

2. **提取关键信息：**
   - 业务背景和目标（这段流程解决什么问题）
   - 参与者角色（谁执行操作？）
   - 核心流程步骤（做什么？按什么顺序？）
   - 决策分支（什么条件下走不同路径？）
   - 状态定义（业务对象有哪些生命周期状态？）
   - 业务规则（有什么约束和校验？）

3. **存档原始输入：**
   将本次读取/接收到的原始需求内容追加到 `design/01-raw-input/{domain-slug}.md`：
   - 如果是文档读取 → 记录文档路径 + 关键段落摘要
   - 如果是终端输入 → 记录对话原文
   - 如果是分块输入 → 每块到达时分别记录
   - 如果是迭代基线 → 记录基线文件版本和修改范围
   - 每条记录标注 `YYYY-MM-DD HH:mm` 时间戳

4. **输出需求理解摘要：**
   向用户复述理解到的业务流程概要，确认是否正确。

### 步骤 2：业务要素提炼分析

**目标：** 将需求转化为结构化业务元素。

对每段需求，提取：

| 要素 | 说明 | 映射到输出的什么部分 |
|------|------|--------------------|
| **角色** | 执行操作的人/系统——从流程中识别出哪些角色参与本领域的业务操作 | 流程图中的节点归属标注 + section 中的角色说明表 + `roles` 数组（文件级结构化数据，含角色→权限点映射） |
| **步骤** | 原子操作或决策 | flowchart node + section.p/ul/ol 文本 |
| **状态** | 业务对象的生命周期状态 | section 中的状态表 + flowchart 中的状态转换节点 |
| **转换** | 从状态 A→B 的触发条件 | flowchart edge label + section 中的规则说明 |
| **规则** | 业务约束和校验 | section 中的 note 块（warning/danger 级别） |
| **异常** | 出错/拒绝/回退时的处理 | section 中的流程分支 + note 块 |
| **关联** | 跨流程/跨域的关联 | section 中的引用链接 |
| **权限点** | 可独立授权控制的业务操作 | `permissions` 数组（文件级结构化数据）+ section 中的权限清单表 |

### 步骤 3：流程建模（含状态机）

**目标：** 构建业务流程的拓扑结构和状态转换图。

#### 3.1 流程图构建原则

| 原则 | 说明 |
|------|------|
| **每个域至少一个 flowchart** | 流程图是业务设计最直观的表达 |
| **状态机推荐 leftright 布局** | 水平展开，左→右阅读 |
| **普通流程推荐 topdown 布局** | 纵向展开，上→下阅读 |
| **每个 flowchat 至少 3 个节点** | 保证有意义的信息密度 |
| **节点 label 使用中文** | 便于业务方理解和审核 |

#### 3.2 节点类型选择

| 业务场景 | 对应 Node type |
|---------|---------------|
| 流程起点 | `start`（绿色体育场形） |
| 流程终点 | `end`（绿色体育场形） |
| 具体操作步骤 | `action`（蓝色圆角矩形） |
| 条件判断 | `decision`（琥珀色菱形） |
| 其他子流程引用 | `subprocess`（紫色双线框） |

#### 3.3 状态机建模

对于涉及状态转换的业务，用一个或多个流程图表达状态机：

- 每个状态 → 一个 node
- 每个转换 → 一个 edge（label 标注触发条件）
- 起点 → `start` node（[*] 初始状态）
- 终止态 → `end` node

同时，在 section 的文本块中补充状态转换表：

```json
{ "type": "table", "headers": ["当前状态", "触发操作", "下一状态", "条件"], "rows": [
  ["draft", "submit", "pending_review", "必填字段齐全"],
  ["pending_review", "approve", "active", "审核通过"],
  ["pending_review", "reject", "draft", "审核不通过"],
  ["active", "suspend", "suspended", "运营手动操作"],
  ["suspended", "restore", "active", "运营手动操作"]
]}
```

#### 3.4 权限控制点提取

**目标：** 从业务流程中提取可独立授权控制的功能点，形成权限点清单。**business-workflow 是权限点的唯一定义源**——API 设计和页面设计阶段只做映射引用，不新增权限点。

**提取原则：**

| 原则 | 说明 |
|------|------|
| **权限即功能** | 权限点由业务功能导出——每个 action 节点对应一个或多个可独立控制的业务操作，不凭空创建 |
| **独立可控** | 一个权限点应能独立授予/撤销，不与其他权限点强绑定 |
| **粒度适中** | 太细（每个字段一个权限）导致管理复杂，太粗（一个权限管全部）失去控制意义。以"用户能完成的一个有意义的操作"为粒度 |
| **角色无关** | 权限点定义时不绑定角色——哪个角色拥有此权限是角色管理阶段的事 |

**提取方法：**

1. 遍历所有 flowchart 中的 `action` 节点：每个 action 都是权限点的候选
2. 合并同类项：多个 action 如果属于同一功能组且总是一起授权，可合并为一个权限点
3. 标注决策节点：`decision` 节点通常不是权限点（它是流程分支条件），但其分支上的 action 各自是权限点
4. 跨域注意：如果某个 action 的操作对象属于其他域，权限点仍在本域定义（谁提供功能谁定义权限），但需标注 `cross_domain` 引用

**权限点在 section 中的表达：**

推荐在对应 section 中使用 `table` block 列出权限清单，同时在文件级 `permissions` 数组中提供结构化定义供下游技能引用。

#### 3.5 角色定义

**目标：** 基于业务流程中识别的参与者角色和已提取的权限点，定义本领域的角色及其权限点聚合关系。

**设计逻辑：** 权限点定义了"能做什么"，角色定义了"谁能做"。在同一个设计阶段把两者都确定下来，业务流程文档就是完整的授权规范——不需要等到 API 设计或实现阶段再去猜测谁该有什么权限。

**角色提取原则：**

| 原则 | 说明 |
|------|------|
| **从流程中来** | 角色从业务流程中自然浮现——观察每个 swimlane/参与者，归纳为角色 |
| **角色与权限点聚合** | 每个角色聚合一组权限点。一个权限点可被多个角色共有 |
| **不绑定具体用户** | 角色定义的是"职位/身份"，不是"张三/李四"。用户与角色的分配是运行时的管理操作 |

**提取方法：**

1. 遍历所有 flowchart 的参与者标注：谁在操作？归纳出角色候选
2. 对每个角色，列出它在流程中执行的所有 action → 映射到已定义的权限点 id
3. 如果角色执行的操作不在已有权限点中 → 说明权限点提取有遗漏，回溯 §3.4 补充
4. 标注角色的 client_group（如 Customer / Staff），用于前端按端分发视图

**角色在 section 中的表达：**

推荐使用 `table` block 列出角色清单（角色名、说明、所属 client_group、聚合的权限点），同时在文件级 `roles` 数组中提供结构化定义。

### 步骤 4：输出生成

**目标：** 将步骤 2 和 3 的结构化数据写入文件。

#### 4.1 数据文件生成规则

- 使用 project 源目录的 `design/02-business-workflow/data/` 路径（非 skill 目录）
- 文件名格式：`{domain-slug}.js`——`{domain-slug}` 从 `design/domain-registry.js` 中获取（含领域编号和英文标识，如 `01-user-identity-permission`）
- 文件内的 `domain` 和 `slug` 字段必须与域注册表中的定义一致
- 数据格式必须精确遵循数据格式 schema（参见下文 §4）
- 同一个域有多个流程图时，使用多个 section，每个含一个 flowchart

#### 4.2 文本内容布局规则

| Block 类型 | 适用场景 |
|-----------|---------|
| `p` | 段落描述、说明文字 |
| `ul` | 列举项、步骤列表 |
| `ol` | 有顺序的步骤、优先级 |
| `table` | 状态转换表、字段说明、角色对照 |
| `note(warning)` | 重要约束、注意事项 |
| `note(danger)` | 关键风险、不可逆操作提醒 |
| `note(info)` | 补充说明、原理阐述 |
| `hr` | 语义分割 |

#### 4.3 与后续步骤的关联

在输出生成时，标注该工作流与后续流水线步骤的关系：
- 哪些实体和字段需要进入 ER 设计
- 哪些状态转换需要 API 端点支持
- 哪些业务规则需要在 API 层校验

### 步骤 5：迭代修正

**目标：** 根据用户反馈持续更新工作流设计。

1. **存档修正输入：** 按步骤 1 的存档规则（追加到 `design/01-raw-input/{domain-slug}.md`，标注时间戳），将用户的修正意见原文存档。

2. **域注册表同步：** 如果在设计过程中发现需要调整领域边界（新增/合并/拆分/重命名领域），**必须按 development-standard skill §5.3 的规则操作：先更新 `design/domain-registry.js`，再更新本数据文件**。不允许先改自己的数据再考虑是否更新注册表

3. **增量更新：** 用户提出修正意见后，定位到对应的 section/flowchart
4. **局部修改：** 只修改受影响的 blocks 或 flowchart nodes/edges，不重写整个文件
5. **版本记录：** 在数据文件的 `last_updated` 字段更新日期
6. **上下文保持：** 每次迭代保持之前的设计决策和历史记录
7. **同步 viewer：** 如果项目有 `workflow-viewer.html`，同时更新其 ALL_DATA 内联数据
8. **权限点反推接收：** 当下游技能（api-design、mobile-app-design、desktop-ui-design）在映射过程中发现权限点缺失或不完善时，会反推回 business-workflow 要求补充。接收反推后按本步骤正常流程处理——存档、定位、修改。如果反推的权限点归属存在歧义（如不确定该权限应属于哪个域、或粒度是否合适），提示用户进行仲裁

## 4. 输出格式（精确 Schema）

> 以下定义输出数据文件的精确格式。所有输出必须严格遵循此 schema。

### 4.1 文件级结构

```javascript
window.WF_DATA = window.WF_DATA || {};
window.WF_DATA["slug-matching-loader-filename"] = {
  "domain":        string,   // 领域编号，如 "01"
  "title":         string,   // 领域中文名，显示为页面主标题
  "slug":          string,   // URL slug，kebab-case
  "description":   string,   // 领域描述，显示在标题卡片副标题
  "last_updated":  string,   // 日期，如 "2026-06-14"
  "sections":      Section[] // 章节列表
};
```

### 4.2 Section 结构

```jsonc
{
  "id":        string,    // 唯一标识，用于 URL 锚点和目录跳转
  "title":     string,    // 章节标题，显示为 <h2>/<h3> + 左侧目录条目
  "level":     number,    // 1/2/3，控制标题级别和目录缩进
  "blocks":    Block[],   // 文本块列表（可选）
  "flowchart": Flowchart  // 流程图定义（可选）
}
```

### 4.3 Block 类型（8 种）

| type | 渲染结果 | 必需字段 | 示例 |
|------|---------|---------|------|
| `p` | `<p>` 段落 | `text` | `{"type":"p","text":"..."}` |
| `h2`/`h3`/`h4` | 子标题 | `text` | `{"type":"h3","text":"标题"}` |
| `ul` | `<ul>` 无序列表 | `items: string[]` | `{"type":"ul","items":["a","b"]}` |
| `ol` | `<ol>` 有序列表 | `items: string[]` | `{"type":"ol","items":["1","2"]}` |
| `note` | 彩色提示框 | `text`, `level`("info"/"warning"/"danger") | `{"type":"note","level":"warning","text":"注意"}` |
| `table` | `<table>` 表格 | `headers: string[]`, `rows: string[][]` | `{"type":"table","headers":["列1"],"rows":[["值"]]}` |
| `hr` | `<hr>` 分隔线 | 无 | `{"type":"hr"}` |

### 4.4 Flowchart 结构

```jsonc
{
  "layout": "topdown",  // 或 "leftright"（状态机推荐 leftright）
  "nodes": Node[],
  "edges": Edge[]
}
```

### 4.5 Node 类型（5 种）

| type | SVG 形状 | 颜色 | 用途 |
|------|---------|------|------|
| `start` | 体育场形（大圆角矩形） | 绿色 `#10b981` | 流程起点 |
| `end` | 体育场形（大圆角矩形） | 绿色 `#10b981` | 流程终点 |
| `action` | 圆角矩形 | 蓝色 `#3b82f6` | 操作/步骤 |
| `decision` | 菱形 | 琥珀色 `#f59e0b` | 条件分支 |
| `subprocess` | 双线框矩形 | 紫色 `#8b5cf6` | 子流程引用 |

每个 Node 必有字段：`id`(唯一标识), `type`(上述5种之一), `label`(显示文本，支持 `\n` 换行)

**Node 数据协议字段（可选，强烈推荐）：**

> 以下字段用于正向追溯和反向验证（见 development-standard §2.6、§2.7）。初始设计时可为空，§8.9 和复查阶段必须补齐。

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `inputs` | `string[]` | 本节点需要的输入数据（字段名），来源是上游节点的 outputs 或外部输入 | `["tier_id", "venue_id"]` |
| `outputs` | `string[]` | 本节点产出的数据（字段名），供下游节点消费 | `["seat_id", "tier_id", "price"]` |
| `consumers` | `string[]` | 哪些下游节点的 id 消费了本节点的 outputs | `["create_order", "generate_pass"]` |

**数据协议验证规则：**
- 每个 `action` 节点的 `outputs` 如果非空，必须至少被一个下游 action 的 `inputs` 引用
- 如果 `outputs` 中某个字段没有任何 `consumers`，标记为 ⚠️ orphan_output
- 如果 `inputs` 中某个字段无法从任何上游节点或外部源获取，标记为 🔴 missing_source

### 4.6 Edge 结构

```jsonc
{ "from": "源节点id", "to": "目标节点id", "label": "边标签" }
```

`label` 可选，显示在连线中点旁。

### 4.7 流程图布局

使用 **Sugiyama 分层布局算法**（最长路径分层 + Barycenter 交叉最小化 + 贝塞尔曲线边）。
只需定义 nodes 和 edges 的**拓扑关系**，坐标由算法自动计算。

### 4.8 权限点定义（可选但推荐）

文件级 `permissions` 数组，为下游 API 设计、页面设计提供结构化权限引用：

```jsonc
"permissions": [
  {
    "id":       string,   // 权限常量名（唯一标识），PascalCase，如 "ArticlePublish"
    "name":     string,   // 中文名，如 "发布文章"
    "desc":     string,   // 说明：该权限允许用户做什么
    "category": string,   // 功能分类（可选），如 "content_management"
    "section_refs": string[]  // 关联的 section id 列表（可选）
  }
]
```

| 字段 | 必需 | 说明 |
|------|------|------|
| `id` | 是 | 权限常量名，PascalCase。API 设计和页面设计通过此 id 引用权限点。这是权限点的**逻辑标识**，与实现细节解耦 |
| `name` | 是 | 中文名，便于业务方理解 |
| `desc` | 是 | 一句话说明该权限允许的操作 |
| `category` | 否 | 功能分组，便于权限管理界面按组展示 |
| `section_refs` | 否 | 关联的 section id，用于追溯权限点的业务来源 |

> **设计逻辑：** `permissions` 是 `sections` 中权限清单表的**结构化等价物**。section 中的 table 供人阅读，`permissions` 数组供下游技能（api-design、mobile-app-design、desktop-ui-design）做权限映射引用。两者内容应保持一致。
>
> **权限模型：** 本平台采用 RBAC（基于角色的访问控制）。角色（Role）聚合一组权限点（以 `id` 字符串集合存储），用户通过被分配角色获得权限。鉴权中间件直接校验用户角色集合中是否包含端点所需的权限点 `id`。不做位掩码运算——避免了权限数量超出固定位长的风险。

### 4.9 角色定义（可选但推荐）

文件级 `roles` 数组，定义本领域涉及的角色及其权限点聚合关系：

```jsonc
"roles": [
  {
    "id":            string,    // 角色标识，PascalCase，如 "Editor"
    "name":          string,    // 中文名，如 "内容编辑"
    "desc":          string,    // 角色说明
    "client_group":  string,    // 所属客户端组，如 "Customer" | "Staff"
    "permission_ids": string[]  // 该角色聚合的权限点 id 列表（引用 permissions[].id）
  }
]
```

| 字段 | 必需 | 说明 |
|------|------|------|
| `id` | 是 | 角色标识，PascalCase。API 鉴权和前端页面组件显隐判断时使用 |
| `name` | 是 | 中文名，便于业务方理解 |
| `desc` | 是 | 一句话说明该角色的职责范围 |
| `client_group` | 是 | 所属客户端组，决定该角色在哪个端（小程序观众端/拳手端/运营后台）登录和操作 |
| `permission_ids` | 是 | 该角色聚合的权限点 id 列表，引用本文件 `permissions` 数组中的 `id`。一个权限点可被多个角色共有 |

> **设计逻辑：** 角色定义和权限点定义在同一个文档中闭环——`roles[].permission_ids` 引用 `permissions[].id`，形成完整的 RBAC 授权规范。API 鉴权中间件和前端权限控制均以此为准。
>
> **跨域角色：** 如果一个角色跨多个领域操作（如 Staff 在多个域都有权限），该角色在**每个相关领域**的 `roles` 中分别定义，但 `permission_ids` 只列出**本域**的权限点。全局角色聚合由鉴权模块在运行时处理。

## 5. 完成检查清单

每次设计迭代完成后，对照以下清单检查：

| 检查项 | 要求 |
|--------|------|
| 数据文件语法正确 | 符合 §4 的数据格式 schema |
| 流程图存在 | 至少 1 个 flowchart，节点 ≥ 3 个 |
| 文本内容完整 | sections 覆盖了所有业务步骤 |
| 状态机已定义 | 涉及状态的域有状态转换表或流程图 |
| 业务规则已记录 | 约束和校验以 note/table 形式记录 |
| 角色已识别 | 流程中的参与者是否已归纳为角色，每个角色的职责是否明确 |
| 角色已定义 | `roles` 数组是否已定义，id/name/desc/client_group 是否完整 |
| 角色↔权限点映射 | `roles[].permission_ids` 是否完整覆盖了该角色在流程中执行的所有操作对应的权限点 |
| 异常流程已覆盖 | 失败/取消/回退的情况已处理 |
| 权限点已提取 | 每个 action 节点是否已纳入权限点清单，是否有遗漏的可授权操作 |
| 权限点结构化 | `permissions` 数组是否已定义，id/name/desc 是否完整 |
| 数据协议已标注 | action 节点的 `inputs`/`outputs`/`consumers` 是否已填写（初始可为空数组，但不可缺失） |
| 数据流无断裂 | 每个 action 的 `outputs` 是否被下游 action 的 `inputs` 消费；`inputs` 是否都能追溯到上游 `outputs` 或外部源 |
| 迭代历史保留 | 未丢失之前的设计决策 |

## 9. 设计原则

- **每个域至少包含一个 flowchart**——流程图是业务设计最直观的表达
- **状态机推荐 `leftright` 布局**（水平展开），普通流程推荐 `topdown`（纵向展开）
- **section.level=1** 为顶级章节（加粗标题），level=2/3 为子章节（缩进）
- **修改数据文件后刷新浏览器即可**——无需任何构建、转译、打包步骤
- **跨项目复用只需两步**：复制 workflow-viewer.html + 创建 data/ 目录
- **必须先出计划再执行**——每次调用先输出步骤计划，用户确认后开始

## 1. 项目初始化

当用户说"新建项目"或"初始化项目"时，按以下流程执行。

### 1.1 收集项目信息

首先与用户对话收集以下信息：

| 信息 | 必需 | 默认值 | 说明 |
|------|------|--------|------|
| **项目名称** | ★★★ | — | 中文全称 |
| **项目描述** | ★★★ | — | 一句话说明 |
| **项目标识（slug）** | ★★☆ | 由名称自动生成 | kebab-case，用于目录命名 |
| **端类型** | ★★★ | — | 一个或多个：微信小程序、App、Web 管理后台、PC 桌面端 |
| **后端技术栈** | ★★☆ | ASP.NET Core + EF Core | 可选覆盖 |
| **前端测试工具** | ★★☆ | Jest | 可选覆盖 |
| **后端测试命令** | ★★☆ | `dotnet test` | 可选覆盖 |
| **目标用户** | ★☆☆ | — | 谁会用这个系统 |

> 技术栈默认值与第二部分 §11.2 一致。用户选默认值时 CLAUDE.md 标注"与标准一致，无覆盖"。

### 1.2 创建目录结构

根据收集到的信息创建以下目录（`{project-slug}` = 项目标识）：

```
{project-slug}/
├── CLAUDE.md                          ← 从模板生成
├── docs/
│   └── development-standard.md        ← 从本 skill 第二部分提取，写入单独文件便于查阅
├── design/
│   ├── domain-registry.js             ← 从模板生成
│   ├── 01-raw-input/                  ← 原始交互存档（瀑布流第一步）
│   ├── 02-business-workflow/data/     ← 工作流数据（§8.1）
│   ├── 03-entity-relationship/data/   ← ER 数据（§8.2）
│   ├── 04-platform-api/domains/       ← API 设计文档（§8.3）
│   ├── 05-{client-slug}/data/         ← 各端页面设计（§8.4，按用户选的端类型创建）
│   ├── 06-tdd/                        ← TDD 测试设计文档（§8.5）
│   └── review/                        ← 复查报告
```

**端类型与目录名映射：**

| 用户说的端 | client-slug | 渲染器由哪个 skill 创建 |
|-----------|------------|----------------------|
| 微信小程序 | `miniprogram` | `mobile-app-design` |
| App | `app` | `mobile-app-design` |
| Web 管理后台 | `web-admin` | `desktop-ui-design` |
| PC 桌面端 | `desktop` | `desktop-ui-design` |

> 渲染器文件（`workflow-viewer.html`、`er-viewer.html`、`design-viewer.html`）
> 由各 skill 的"首次搭建"步骤负责复制。本 skill 只创建空目录骨架。

### 1.3 生成 CLAUDE.md

使用 `templates/CLAUDE.md`，替换占位符：

| 占位符 | 替换为 |
|--------|-------|
| `{{project-name}}` | 项目名称 |
| `{{project-description}}` | 项目描述 |

从本 skill 第二部分（标准全文）提取内容，写入新项目的 `docs/development-standard.md`，

### 1. 概述

#### 1.1 这是什么

这是一套 **从实践中长出来的软件开发方法**，而非纯理论推导。它起源于一个真实项目的多轮迭代（业务设计 → ER → API → 页面 → 代码 → 一致性校核 → 修正），在经历了 60+ 项细节打磨后凝结而成。

#### 1.2 适用场景

| 特征 | 说明 |
|------|------|
| **团队规模** | 单人开发者或 2-3 人极小型团队。不预设多人协作的同步/协商机制 |
| **项目类型** | 从零开始的绿地项目，或可重构的存量项目 |
| **技术栈** | 以 .NET + 小程序为默认栈，但方法论可迁移（详见各步骤的默认值标注） |
| **AI 协作** | 假定开发全程有 AI 辅助（如 Claude），方法中包含针对 AI 认知局限的对冲机制 |

#### 1.3 版本与演进

本标准自身通过实战迭代演进。每次在项目中发现问题后：

| 问题类型 | 处理方法 |
|---------|---------|
| 标准未覆盖的方法论空白 | 修订标准文档（即本 skill 第二部分） |
| 标准已覆盖但执行不到位 | 修复实现，标准不变 |
| 标准不合理/可优化 | 经实践验证后修订标准 |

### 2. 核心理念

#### 2.1 设计资产优于代码资产

代码是最终产物，但设计资产（工作流、ER 图、API 文档、页面功能树）定义了代码的上限。**先有可验证的设计，后有可执行的代码。**

#### 2.2 结构化数据驱动文档

文档不是写给人看的大段文字，而是结构化的数据（JSON / JS 对象）。数据即真相源，渲染层可替换。这使得：
- 文档可以被 AI 精确读取和 diff
- 同一数据可用多个视图展示（树形、图形、表格）
- 文档本身可作为自动化检查的输入

#### 2.3 契约先行

API 设计文档是前后端之间、人与 AI 之间的核心合约。**合约定稿后再实现**，避免"边写边商量"导致的实现偏差。

#### 2.4 认知偏差防御

AI 生成的代码 + AI 写的测试 = 同一套认知模型的产物。如果对需求的理解存在盲区，这个盲区会同时污染代码和测试——代码写错了，测试也按错误理解验证，全部绿色但上线就炸。因此需要**不同视角的对抗审查**来打破认知回音室。

#### 2.5 决策可追溯

每一个设计结论都可以回溯到原始的"为什么"——用户说了什么、基于什么理由做了这个决定。这是通过**原始交互存档**（第7节）实现的。

#### 2.6 正向定义——每一步的输出必须可穷举验证

流水线的每一步，产出物不能只是"描述性文档"。每个步骤的输出必须包含**可穷举验证的数据协议**：

| 步骤 | 产出物 | 必须包含的数据协议 |
|------|--------|-------------------|
| §8.1 工作流 | flowchart node | `inputs`（字段级输入来源）+ `outputs`（字段级产出）+ `consumers`（谁消费这些产出） |
| §8.2 ER | field | `source`（被哪个工作流节点产出 / 哪个原始需求要求）+ `consumers`（被哪些 API 消费） |
| §8.3 API | endpoint | `consumes`（请求字段来源追溯）+ `produces`（响应字段去向追溯） |
| §8.4 页面 | page node | `page_input`（从哪个页面进来 + 接收什么参数）+ `page_output`（去向哪个页面 + 传递什么参数） |
| §8.4 页面 | component node | `api_ref`（调用哪个 API）+ `sends`（发送什么数据给 API） |

**核心规则：步骤 B 消费步骤 A 的产出时，如果 A 的输出中缺少 B 需要的字段，则 A 的产出不合格——步骤 B 必须拒绝启动并报告缺失项。**

以上数据协议字段在初始设计阶段为**可选**（初始设计时可能无法确定完整的上下游），但 §8.9 端到端验证和复查阶段必须检查：无数据协议的节点标记为"未追溯"，跨步骤数据流断裂标记为断点。

#### 2.7 反向验证——从最终产物反推需求是否落地

正向设计完成后，必须执行反向验证。这不是"再加一个步骤"，而是**嵌入每个步骤的完成检查清单中**：

**反向验证方法：**
1. 从当前步骤的产出中，选一条用户旅程（如"观众购票"）
2. 从旅程的终点（如"电子票详情页"）反推：
   - 这个页面显示的数据从哪里来？（API → ER → 工作流节点 → 原始需求）
   - 每一步的字段是否能逐层对上？
3. 从旅程的起点正推：
   - 原始需求说"用户选座"→ 工作流有选座节点吗？→ ER 有座位实体吗？→ API 有选座端点吗？→ 前端有选座页面吗？→ 选座页产出的 seat_id 能传到订单 API 吗？→ 电子票上能显示座位号吗？
4. 任一环节中断 → 报告断点，阻断当前步骤的完成

**正向和反向都必须走完每一个环节，不能跳过。**

### 3. 设计原则

| 原则 | 说明 |
|------|------|
| **结构化数据优先** | 设计资产以 JSON/JS 等结构化格式存储，便于 AI 精确读取、diff 和验证 |
| **单一渲染器** | 同一类设计资产共用一套渲染逻辑，样式统一，维护成本低 |
| **不重复定义** | 每个概念（实体、字段、规则）只在归属域定义一次，其他域通过引用使用 |
| **可点击导航** | 跨域/跨资产的关系是超链接，不是静态引用——从任意资产可跳转到关联资产 |
| **无外部依赖** | 设计资产渲染零外部依赖（不依赖 Drawio、Mermaid、Visio 等第三方工具或 JS 库） |
| **版本控制友好** | 所有设计资产以纯文本存储，可 diff、可 review、可回滚 |

> 技能定义文件位于 `.claude/skills/{skill-name}/SKILL.md`（项目级）或 `~/.agents/skills/{skill-name}/SKILL.md`（全局发布版）。

#### 4.1 Skill 设计约束

所有对外产出数据的设计或代码生成 skill，必须同时具备两种执行模式：

| 模式 | 说明 | 示例 |
|------|------|------|
| **创建模式** | 从零生成设计资产或代码骨架 | `business-workflow` 创建新的工作流数据 |
| **一致性检查模式** | 以已有资产为输入，校验与上游产物的一致性，输出结构化问题清单 | `mobile-app-design` 检查 tree.js 与 ER/工作流的一致性 |
| **调度模式** | 不自行执行比对，而是逐维度委托对应 skill 执行检查，汇总结果 | `review` 委托各 skill 做深度检查 |

> **设计意图：** review 不应亲自比对代码、字段或路由——那是各专业 skill 的工作。review 是**纯调度器**，逐维度调用对应 skill 的一致性检查模式，只做问题汇总和交互管理。

**一致性检查职责的完整委托表见 `review` skill。** 各 skill 的检查模式在各自 SKILL.md 中定义。

### 5. 通用约定

#### 5.1 Commit 规范

- **Commit message 必须使用中文书写**。提交代码时，commit message 用中文简要描述变更内容，格式不限，但必须为中文。

#### 5.2 项目根目录三层结构

#### 5.3 域注册表（Domain Registry）

`design/domain-registry.js` 是项目所有业务领域定义的**中心注册表**，是流水线执行过程中持续维护的动态设计资产。它在初始域范围界定阶段创建，随后在七步流水线的执行过程中逐步完善。

**格式：**

```javascript
PS_DATA.domainRegistry = {
  identity: { project: '项目名', standard: 'SFDS v1' },
  domains: [
    { id: '01', slug: 'user-identity',   name: '用户身份',   description: '...' },
  ]
};
```

**读写规则（Read/Write Semantics）：**

| 规则 | 说明 |
|------|------|
| **先读后写** | 任何流水线步骤在执行前，必须从 `domain-registry.js` 读取当前领域定义；`{domain-slug}` 参数以此文件为准 |
| **改域先写注册表** | 当任何步骤发现领域边界需要调整（合并、拆分、重命名、新增、删除领域），**必须先更新 `domain-registry.js`，再更新自身数据文件**。不允许"先改自己的数据，再考虑是否更新注册表" |
| **注册表优先** | 注册表是领域定义的唯一权威来源。任何步骤发现自己的数据与注册表不一致时（如 slug 拼写不同、领域边界不同），以注册表为准，修正自身数据 |
| **变更即事件** | 领域边界变更是全局事件。修改注册表后，所有依赖于受影响领域的后续步骤在下一次执行时必须重新同步。迭代循环（§10）天然支持这一机制——一致性校核会检测到 slug 不一致 |

> **设计意图：** 域注册表承担着所有设计 skill 之间的 **同步锁（synchronization lock）** 角色。它把"领域有哪些"这个事实从各个 skill 的隐式缓存中提取出来，变成显式的、可 diff 的、单人可维护的单一事实源。

#### 5.4 Skill CHANGELOG 规范

所有对外产出数据的 skill 必须在每次变更后将变更记录写入其负责目录下的 `CHANGELOG.md`。具体路径和格式由各 skill 自行定义。这是强制要求，不是建议。

#### 6.4 设计决策日志（design-decisions.js）

> **定位：** 跨步骤共享的决策记录。当 AI 在流水线任一步骤做出设计选择（不只是"从上游推导"，而是"在多个方案中选择了这个"），必须写入此文件。
> **作用：** 解决"从原始需求重新推导时，AI 不知道中间发生了什么决策"的问题。

**格式：**

```javascript
window.DESIGN_DECISIONS = [
  {
    id: "DD-001",
    date: "2026-06-22",
    domain: "ticketing",           // 所属领域，跨域决策用 "cross-domain"
    layer: "api-design",           // 在哪个步骤做出的决策
    trigger: "implementation-feedback",  // user-decision | upstream-change | implementation-feedback | review-finding
    summary: "退款端点使用 Idempotency-Key 而非数据库唯一约束来防止重复退款",
    rationale: "代码实现验证了数据库唯一约束在并发场景下不可靠，Idempotency-Key + Redis 缓存提供了更强的保证",
    affected_assets: [
      "design/04-platform-api/data/ticketing.js",
      "design/04-platform-api/backend-architecture/data/resilience-policy.js"
    ],
    alternatives_considered: ["数据库唯一约束", "分布式锁"],
    supersedes: []                  // 替代了之前的哪个决策 ID
  }
];
```

**写入规则：**

| 规则 | 说明 |
|------|------|
| **任何非推导性选择必写** | 如果设计结论不能从上游资产纯逻辑推导出来（涉及权衡、偏好、经验判断），必须写入 |
| **实现反馈必写** | 当代码实现发现比设计更优的方案，通过同步螺旋回写设计时，必须记录决策 |
| **不可删除历史决策** | 旧决策标记 `superseded_by: "DD-XXX"`，不物理删除 |
| **所有 skill 可读写** | `api-design`、`backend-architecture-design`、`api-code-gen`、`tdd-build` 等均可写入 |

**读取规则：**
- 同步螺旋执行 Step 2（上溯）时，先查设计决策日志——不一致可能是**有意为之**的决策，不是错误
- 从原始需求重新推导时，先加载设计决策日志——知道中间做出了哪些选择，推导时保留这些选择

#### 6.5 资产同步状态追踪（sync-status.js）

> **定位：** 位于 `design/review/sync-status.js`，由 review skill 在每次复查后更新。记录每个设计/代码资产的版本和同步状态。
> **作用：** 让 AI 一眼知道哪些资产对之间不同步，避免每次全量扫描；增量同步时只处理 dirty 资产。

**格式：**

```javascript
window.SYNC_STATUS = {
  last_sync_pulse: "2026-06-22T15:30:00",
  last_review: "2026-06-22-review-v3",
  assets: [
    {
      path: "design/03-entity-relationship/data/ticketing.js",
      version: 3,
      synced_with: [
        { target: "design/04-platform-api/data/ticketing.js", version: 5 },
        { target: "src/server/Models/TicketOrder.cs", version: 3 }
      ],
      dirty: false
    },
    {
      path: "design/04-platform-api/backend-architecture/data/resilience-policy.js",
      version: 2,
      synced_with: [
        { target: "design/04-platform-api/data/ticketing.js", version: 5 }
      ],
      dirty: true,
      dirty_reason: "API 新增了退款端点但架构幂等列表未同步",
      dirty_since: "2026-06-22T14:00:00"
    }
  ]
};
```

**更新规则：**
- review 每次复查完成后，更新 `sync-status.js`
- 同步螺旋每次收敛后，将涉及资产的 `dirty` 标记为 `false`
- 任何 skill 修改资产后，将该资产标记为 `dirty: true`（在输出的同时更新）

### 7. 横切关注点：原始交互存档

#### 7.1 定位

> **这不是一个独立的步骤，而是贯穿所有步骤的横切任务。** 流水线中没有一步叫"原始内容存档"——但每一步都必须执行存档。

#### 7.2 规则

设计过程中所有用户的自然语言交互内容——原始需求描述、设计决策讨论、修改意见、反馈纠错——以**原始文本形式**存档。

| 规则 | 说明 |
|------|------|
| **存放位置** | `design/01-raw-input/{domain-slug}.md` |
| **文件粒度** | 每个领域一个文件；跨领域全局讨论放在 `design/01-raw-input/global-design-discussion.md` |
| **追加方式** | 同一领域的多次交互追加到同一个文件，新内容加在末尾 |
| **记录格式** | 每次交互注明 timestamp + 内容原文 |
| **覆盖范围** | 原始需求、设计讨论、修改意见、迭代反馈、决策变更 |
| **不做什么** | 不修改原文、不提炼摘要、不归纳分类 |

#### 7.3 存档时机

| 场景 | 存档内容 |
|------|---------|
| 用户提出新的业务需求 | 需求描述的原文 |
| 用户对工作流设计提出修改 | 修改意见和理由 |
| 用户指出 ER 中遗漏的字段 | 指出的内容和业务场景 |
| 用户要求调整 API 参数 | 调整要求和约束条件 |
| 团队讨论后的决策 | 决策依据和结论 |

### 8.0 执行顺序与依赖关系（强制约束）

> **⚠️ 以下依赖关系是强制约束，不是建议。** 各步骤对应的 skill/Agent 必须等待其所有前置步骤完成后方可启动。

#### 依赖关系定义（纯文本逻辑——供 AI Agent 解析）

**串行链（必须按序执行）：**

1. §8.1 business-workflow — 无前置，流水线起点。输入：raw-input + domain-registry
2. §8.2 entity-relationship — 依赖 §8.1 完成。输入：§8.1 产出 + raw-input + domain-registry
3. §8.3 api-design — 依赖 §8.1 + §8.2 完成。输入：§8.1+§8.2 产出 + raw-input + domain-registry
4. §8.4 页面设计 — 依赖 §8.3 完成。输入：§8.3+§8.2+§8.1 产出 + raw-input + domain-registry
5. §8.5 tdd-build — 依赖 §8.3 + §8.4 完成。输入：§8.3+§8.4+§8.1+§8.2 产出 + domain-registry

**并行区间（两条独立分支，可同时执行）：**

6. §8.3b 后端架构 — 依赖 §8.3 完成。输入：§8.3+§8.2+§8.1 产出
7. §8.4 页面设计 — 依赖 §8.3 完成。输入：§8.3+§8.2+§8.1 产出

> §8.3b 与 §8.4 可并行执行——两者同依赖 §8.3，互不依赖。

8. §8.5 tdd-build — 依赖 §8.3+§8.4 完成即可启动（API 测试 + 前端测试）；架构承诺测试和契约测试在 §8.3b 完成后增量补充。
   > **关键：** TDD 不再被 §8.3b 阻塞。前端测试只需要页面设计（§8.4），不需要后端架构（§8.3b）。

> §8.6 与 §8.7 可并行执行。

**测试执行（§8.6 + §8.7 全部完成后的独立步骤）：**

11. §8.8 tdd-execute — 依赖 §8.6 + §8.7 全部完成。输入：§8.5 测试代码 + §8.6 API代码 + §8.7 前端代码

> 这避免了"测试写好了但 API 代码还没生成 → 因缺失实现而失败 → 失败没有分析价值"的无意义失败。

**最终验证：**

9. §8.9 e2e — 依赖 §8.8 全部完成（预留）

| 前端设计 → 前端代码（桌面端） | desktop-code-gen | 设计→代码一致性检查模式 |
| API → 前端 API 调用 | review（委托 mobile-code-gen / desktop-code-gen） | API 路径+结构校核 |
| TDD 设计 → TDD 代码 | tdd-build | 用例覆盖+断言一致性检查（一致性检查模式） |
| 测试代码 → API 代码 | tdd-execute | 实际执行测试+失败溯源+循环修复至全绿 |

各 skill 的检查模式通过在对应 skill 中定义的输出格式（结构化 JSON）相互解耦，
review skill 负责任何调度和汇总。

- **下游:** §8.9 e2e（预留）+ Review 复查门控

> **⚠️ 本步骤是流水线的关键质量关口。** 测试必须全部通过（0 失败）方可进入 Review 或提交代码。
> 不得仅凭 TDD 报告中"全部通过"的声明跳过实际执行——必须亲自运行 `dotnet test` 并记录结果。
> 测试执行中发现的设计层问题必须回溯到对应 §8.1~§8.4 步骤修正，不可在 §8.8 中直接修改设计文件。

#### 8.9 关键旅程冒烟验证（必做，不可跳过）

> **触发条件：** §8.8 tdd-execute 全量通过后自动进入。
> **与 §8.8 的区别：** §8.8 验证 API 合约（请求→响应），§8.9 验证用户体验路径（需求→页面→API→页面→结果）。

**执行方式：**

1. **旅程提取：** 从域注册表中为每个域选取 2-3 条最高优先级用户旅程。旅程定义来自 business-workflow 的流程图（从 start 节点走到 end 节点的完整路径）。

2. **逐旅程走查：** 对每条旅程，从起点页面走到终点页面，逐步检查：

   | 检查项 | 方法 | 失败标记 |
   |--------|------|---------|
   | 页面文件齐全 | 检查 js/wxml/wxss/json 四件套存在 | 🔴 page_missing |
   | API 路径可调通 | 请求每个旅程涉及的 API，确认返回 2xx；**并验证响应 JSON 中是否包含 API 设计 `produces[]` 声明的关键字段**（至少验证 `produces` 中标注 `consumer` 的字段实际出现在响应中） | 🔴 api_missing / 🔴 field_missing |
   | API 响应格式匹配 | 对每条旅程的关键 API，对比实际响应 JSON 的字段结构与 API 设计文档的 `responses[].fields` 定义是否一致。字段名、类型、嵌套层级差异标记 | ⚠️ format_mismatch |
   | 页面间数据传递完整 | 检查 `page_output.params[]` 的每个字段是否全部出现在下游页面的 `page_input.params[]` 中 | 🔴 data_flow_broken |
   | 组件-API 绑定正确 | 检查组件的 `api_ref` 指向的端点存在，且 `sends[]` 字段匹配 API 的 `consumes[]` | 🔴 api_mismatch |
   | 旅程终点数据可反推 | 旅程最终产出的数据（如 ticket 上的 seat_id）能通过 `produces → consumers` 链反推回原始需求 | 🔴 untraceable |

3. **输出：** `design/review/smoke-report.md`，每条旅程 1 行状态（✅ 全通 / 🔴 断点位置 + 缺失详情）。

4. **退出条件：** 所有关键旅程无 🔴 断点。⚠️ 警告项（如数据协议缺失但当前步骤可走通）必须录入 `design/review/remaining-issues.md`。

#### 8.9b 真实浏览器端到端验证（可选，推荐）

> **与 §8.9 的关系：** §8.9 是文件/API 级静态走查，§8.9b 是用真实浏览器（Playwright）自动化执行用户旅程。
> **触发条件：** §8.9 通过后自动进入。如果项目未配置 Playwright 或不涉及 Web/小程序端，可跳过此步骤。
> **适用范围：** Web 管理后台、H5 页面、任何可通过浏览器访问的端。

**执行方式：**

1. **环境检查：** 确认 Playwright 可用（`npx playwright --version`）。若不可用，记录跳过原因，不阻断流水线。

2. **旅程脚本化：** 将 §8.9 的关键旅程转化为 Playwright 脚本：
   - 每条旅程 = 一个 `test()` 块
   - 每一步 = page navigation / click / fill / assert
   - 关键断言：页面可加载（无 pageerror）、点击后导航到正确 URL、表单提交后 API 返回成功、关键数据（如 ticket seat_label）在 DOM 中可见

3. **执行：** `npx playwright test --project=chromium`。不可用的端（如微信小程序）跳过。

4. **输出：** Playwright 测试报告（HTML + JSON），追加到 `test-reports/` 目录。

5. **退出条件：** 可执行的旅程全部通过。不可执行的环境/端 → 记录跳过，不阻断。

### 9. 复查门控机制

#### 9.1 认知基础

TDD 验证"行为是否正确"，复查验证"设计是否合理、代码是否可维护、是否有 TDD 测试盲区"。

> **关键认知：** AI 生成的代码 + AI 写的测试 = 同一套认知模型的产物。需要不同视角的对抗审查打破认知回音室。

#### 9.2 触发点

| 触发点 | 时机 | 执行者 |
|--------|------|--------|
| **A: API 代码复查** | §8.6 完成后 | `review` skill 调度各检查维度 |
| **B: 前端代码复查** | §8.7 代码生成后 | `review` skill 调度各检查维度 |
| **C: 全量复查（强制）** | §8.8 tdd-execute 全绿后 | `review` skill 调度全部 11 个校核维度 |

> **说明：** 触发点 A/B 是轻量检查（代码结构、编译、API 签名对齐），触发点 C 是完整的全维度复查。
> 触发点 C 不得在 tdd-execute 通过前启动——测试未全绿时复查代码实现不一致性没有意义。

> 具体检查维度（安全、性能、代码结构、边界条件、错误处理、API 调用正确性、状态管理等）见 `review` skill。

#### 9.3 风险等级与复查策略

| 风险等级 | 典型变更 | 复查策略 |
|---------|---------|---------|
| 🟢 低 | 字段新增、文案修改 | TDD/静态验证即可 |
| 🟡 中 | 新业务逻辑、状态机流转 | 自动化对抗复查 |
| 🔴 高 | 支付、实名认证、数据迁移 | 对抗复查 + 人工确认 |

### 10. 迭代与变更管理

#### 10.1 为什么需要迭代

流水线单次执行几乎不可能达到完美——认知限界、跨域耦合、一致性衰减是系统性原因。

#### 10.2 变更发起权规则

**铁律：设计资产是默认权威源。当代码与设计不一致时，默认修改代码以匹配设计。**

但实现过程中发现的更优方案不应被丢弃——代码层面的发现可以通过以下协议反向更新设计：

| 变更来源 | 可以主动改什么 | 不可以改什么 |
|---------|-------------|------------|
| **设计层面**（§8.1-§8.5，含 §8.3b） | 业务逻辑、数据模型、API 契约、后端架构、页面结构、测试规格 | — 设计是默认权威源 |
| **代码层面**（§8.6-§8.7） | 架构优化、实现模式重构、算法替换、代码风格、Bug 修复 | **业务逻辑 / API 定义 / 页面功能结构 / 数据库 schema** |
| **实现反馈 → 设计** | 当代码实现了比设计更优的方案时，通过 §10.3b「同步螺旋」将代码行为反向记录到设计资产中 | **不可跳过设计更新直接以代码为准长期运行**——代码领先于设计是临时状态，必须尽快通过同步螺旋收敛 |

> **关键变化（v3）：** 旧版规定"代码不能主动变更业务逻辑"——这是对的，但不够。新增"实现反馈路径"：
> 代码层面的优化发现 → 通过同步螺旋协议 → 回写到设计资产 → 设计资产更新后重新成为权威源。
> 代码可以"建议"更好的设计，但不能绕过设计成为长期真相源。

#### 10.3 变更传播规则

设计侧任一环节变更后，下游必须重新验证：

| 我改了… | 正向影响的下游（必须重新验证） | 反向复核的上游 |
|---------|--------------------------|-------------|
| raw-input | 工作流→ER→API→页面→tdd-build→代码→前端→tdd-execute | — |
| §8.1 工作流 | ER→API→页面→tdd-build→代码→前端→tdd-execute | raw-input |
| §8.2 ER | API→页面→tdd-build→代码→tdd-execute | 工作流 |
| §8.3 API | 架构→页面→tdd-build→代码→前端→tdd-execute | 工作流 + ER |
| §8.3b 后端架构 | tdd-build→代码→tdd-execute | API + ER + 工作流 |
| §8.4 页面 | tdd-build→前端代码→tdd-execute | API + ER + 工作流 |
| §8.5 tdd-build | API 代码 + tdd-execute | 架构 + API + ER + 页面 + 工作流 |
| §8.6 代码 | tdd-execute（全量测试执行） | tdd-execute（全绿）+ API 设计（路由/参数/响应一致） |
| §8.7 前端 | tdd-execute（全量测试执行） | 页面设计（功能树对齐） |
| §8.8 tdd-execute | Review 复查 + §8.9 e2e | 失败向上溯源到 §8.6 / §8.3 / §8.1 |

**硬约束（代码变更）：**

| # | 约束 | 触发条件 | 强制动作 |
|---|------|---------|---------|
| 1 | API 代码变更 → TDD 全绿 | 任何 API 代码变更 | 触发 `tdd-execute`（§8.8），全量测试通过方可提交 |
| 2 | ORM 代码变更 → TDD 全绿 | 任何 ORM 代码变更 | 触发 `tdd-execute`（§8.8），全量测试通过方可提交 |
| 3 | TDD 测试代码变更 → 自测通过 | 任何 TDD 测试代码变更 | 触发 `tdd-execute`（§8.8），确认全部通过 |
| 4 | ORM 代码变更 → 数据库同步 | 任何 ORM 代码变更 | 执行项目约定的数据库迁移命令（如 `dotnet ef database update`、`alembic upgrade head` 等） |
| 5 | 启动前 DB ≡ ORM 一致性守卫 | 启动服务器、运行任何测试之前 | 必须先执行项目约定的数据库迁移命令 |

> **说明：** 规则 2（质量守卫）和规则 4（环境同步）均触发于 ORM 变更，但关注点不同。规则 5 更广泛——不管你这次是否改了 ORM，只要跑服务器或测试，DB 必须与代码一致。

**硬约束（设计侧）：**
- 任何设计产出变更（§8.1-§8.4，含 §8.3b）→ 必须通过对应 skill 检查模式验证
- 任何涉及跨域引用的变更 → 必须同步更新 `core-er.js`

#### 10.3b 同步螺旋协议（多向同步）

> **设计意图：** §10.3 定义了单向传播规则（"改了 A → B/C/D 重新验证"）。
> 但在实际迭代中，变更可能从**任意节点**发起——代码优化发现了更好的设计、联调暴露了 API 设计的缺失、
> 实现反馈揭示了原始需求的不清晰。同步螺旋协议确保无论变更起点在哪里，全链条最终收敛到一致状态。

**同步螺旋（Sync Spiral）五步法：**

```
任意节点发生变更（设计 / 代码 / 原始需求 / 测试）
    │
    ├── Step 1 — 标记真相源
    │   确定本次同步的"真相源"节点。
    │   · 代码改好了、设计是旧的 → 代码是真相源
    │   · 需求澄清了、设计还没改 → 需求是真相源
    │   · 设计主动调整了 → 设计是真相源
    │   记录到 design-decisions.js。
    │
    ├── Step 2 — 上溯（真相源 → 上游链路）
    │   真相源的变更是否揭示了上游资产的错误/缺失/过时？
    │   
    │   | 真相源在… | 上溯检查 |
    │   |-----------|---------|
    │   | §8.6 代码 | API 设计的端点/参数/响应是否需要更新？→ 架构设计的策略是否需要更新？→ ER 的约束是否变了？→ 工作流的状态转换描述是否匹配？→ 原始需求是否记录了这个行为？ |
    │   | §8.3 API 设计 | 工作流的 action 是否需要调整？→ ER 的字段是否需要更新？→ 原始需求的场景描述是否清晰？ |
    │   | raw-input | 工作流的 section 是否需要重排？→ 后续全链 |
    │   
    │   上溯规则：
    │   · 每次上溯一层，更新该层资产后，以更新后的资产为新起点继续上溯
    │   · 如果该层资产已经匹配真相源 → 停止该分支的上溯
    │   · 如果上溯到 raw-input → 追加原始交互记录（标注时间戳 + 变更原因），不修改历史原文
    │
    ├── Step 3 — 下传（真相源 → 下游链路）
    │   与 §10.3 的正向传播规则一致，但触发源不限于设计侧。
    │   真相源的变更是否破坏了下游资产？
    │   
    │   | 真相源在… | 下传检查 |
    │   |-----------|---------|
    │   | raw-input | 工作流→ER→API→架构→TDD→代码→前端→测试 |
    │   | §8.3b 架构 | TDD 的架构承诺测试→代码的分层/缓存/幂等实现→tdd-execute |
    │   | §8.6 代码 | tdd-execute（全量测试） |
    │   
    │   下传规则：
    │   · 每次下传一层，更新后以新节点为起点继续
    │   · 如果该层资产已经匹配真相源 → 停止该分支的下传
    │
    ├── Step 4 — 扩散迭代
    │   Step 2 和 Step 3 可能修改了新的节点。
    │   以每个被修改的节点为新起点，重复 Step 2-3，直到：
    │   · 无新变更产生（全链稳定）
    │   · 或连续 2 轮无实质性新差异
    │
    └── Step 5 — 记录与标记
        在 design-decisions.js 中记录本次同步螺旋的：
        · 触发源（哪个节点、什么变更）
        · 螺旋路径（依次修改了哪些文件）
        · 收敛判定（何时稳定）
        在 sync-status.js 中更新所有受影响资产的同步状态为 clean。
```

**同步螺旋的收敛条件：**
- 全链一致性检查通过（委托各 skill 的一致性检查模式）
- §8.8 tdd-execute 全量测试通过
- sync-status.js 中无 dirty 资产

**同步螺旋 vs 标准 §10.3 单向传播的关系：**
- §10.3 回答"改了 A 之后哪些必须重验"（影响面分析）
- §10.3b 回答"无论哪变了，如何让全链回到一致"（收敛方法）
- 两者互补：§10.3 用于判断影响范围，§10.3b 用于执行收敛

#### 10.3c TDD 迭代闭环（设计变更 → TDD 更新 → 验证 → 循环）

> **设计意图：** 设计变更不是"改完就完了"——它必然触发 TDD 的连锁更新。
> §10.3 定义了影响面（"API 变更 → tdd-build 重验"），§10.3b 定义了同步方向，
> 但缺少一个显式的"TDD 怎么跟着改、改了怎么验、不通过怎么回退"的闭环。
> 本节填补这个缺口。

**闭环流程：**

```
设计变更（§8.1-§8.4 任一环节发生变更）
    │
    ├── ① TDD 影响分析
    │     对照变更内容，逐项判断：
    │     · 新增端点 → 需要新增测试用例（正常 + 异常 + 边界）
    │     · 修改响应字段 → 需要更新断言
    │     · 删除端点 → 需要删除或归档对应测试用例
    │     · 修改状态转换 → 需要更新状态机测试
    │     · 修改架构策略 → 需要更新架构承诺测试
    │     输出：受影响用例清单（tc_id 列表）
    │
    ├── ② TDD 设计更新
    │     更新 `design/06-tdd/{slug}-tdd-design.md`：
    │     · 新增用例 → 按模板写 TC-XXX 设计
    │     · 修改用例 → 更新调用序列/断言清单
    │     · 删除用例 → 标记 [已废弃] + 废弃原因 + 日期
    │     · 版本号 +1
    │     检查：设计文档与变更后的上游资产一致
    │
    ├── ③ TDD 代码更新
    │     按更新后的设计文档修改测试代码：
    │     · 新增/修改/删除测试方法
    │     · 更新 Mock/Fixture/TestData
    │     编译通过（0 错误）
    │
    ├── ④ 执行验证（tdd-execute）
    │     dotnet test 全量运行
    │     │
    │     ├── 全部通过（0 失败）
    │     │     └── ✅ 闭环收敛 → 更新 sync-status.js → 进入 Review
    │     │
    │     ├── 部分失败
    │     │     └── 失败分类（按 tdd-execute §2.2）：
    │     │         ├── 测试代码问题（断言错、Mock 错）
    │     │         │     └── 回到 ③ 修正测试代码 → 重新 ④
    │     │         ├── API 实现问题（代码与设计不一致）
    │     │         │     └── 去 §8.6 修正实现 → 重新 ④
    │     │         └── 设计问题（测试揭示设计本身的矛盾）
    │     │               └── 回到 ① 重新分析 → 可能需要回溯上游设计
    │     │
    │     └── 全部失败（编译/环境问题）
    │           └── 检查编译错误、数据库迁移、DI 配置 → 修复后重新 ④
    │
    └── ⑤ 循环终止条件
           · tdd-execute 全绿（0 失败）
           · 且 TDD 设计文档与上游资产一致
           · 且 sync-status.js 中相关资产 dirty=false
```

**与同步螺旋（§10.3b）的关系：**

| 场景 | 用哪个 |
|------|--------|
| 设计主动变更（改了 API/ER/架构）→ TDD 跟着改 | §10.3c TDD 迭代闭环 |
| 代码实现发现更优方案 → 反向更新设计 → TDD 跟着改 | 先用 §10.3b 同步螺旋上溯更新设计，再用 §10.3c 下传更新 TDD |
| 测试执行发现设计矛盾 → 可能需要改设计 | §10.3c 步骤④ 的"设计问题"分支自动触发 §10.3b |

> **关键原则：** 设计变更不是一个原子操作——它是一系列连锁更新的起点。
> TDD 迭代闭环确保这个连锁反应有明确的步骤、分支和终止条件，
> 而不是靠 AI 每次临场发挥。

#### 10.3d 跨级追溯（Cross-Level Traceability）

> **设计意图：** §10.3 的变更传播规则定义的是**相邻层**之间的影响（A→B→C），但经验表明很多设计缺陷藏在**跨级链路**中：
> §8.1 说"用户可以选座"，§8.4 有选座页面，§8.7 有选座代码——但 §8.7 产出的 seat_id 根本没进入 §8.6 的订单 API。
> 相邻层检查全部 ✅，跨级链路却断了。本节强制在复查和 §8.9 阶段执行跨级追溯。

**正向跨级追溯（需求 → 最终产物）：**

| 追溯链 | 检查方法 |
|--------|---------|
| raw-input → workflow → ER → API → page → code | 对每条原始需求，验证全链路中每一层都有对应产物。缺失任一层 → 🔴 break |
| workflow.outputs → API.consumes → page.sends → code | 工作流产出的字段，必须在 API 的请求参数中找到，在页面的 sends 中找到，在代码的 API 调用体中找到 |
| ER.field → API.produces → page.page_output → next_page.page_input | ER 定义的字段，必须在 API 响应中出现，被页面传递，被下游页面接收 |

**反向跨级追溯（最终产物 → 需求）：**

| 追溯链 | 检查方法 |
|--------|---------|
| code → page → API → ER → workflow → raw-input | 每个代码产物（Controller Action、页面文件）必须能反推回至少一条原始需求。反推不到 → ⚠️ orphan |
| page_input.params → page_output.params → API.produces → ER.field | 页面接收的参数，必须能反向追溯到它从哪里产出（上游页面、API 响应、ER 实体） |

**跨级追溯的触发时机：**

| 时机 | 执行者 |
|------|--------|
| §8.9 关键旅程冒烟验证 | 必须执行正向全链追溯（旅程每一步都走完整条追溯链） |
| Review D10 跨级追溯校验 | 对域内所有核心实体，各选一条全链做正向+反向追溯 |
| 用户提出"XX 功能为什么没有" | 执行该功能的正向追溯，定位第一个缺失层 |

**与 §10.3 单向传播的关系：**
- §10.3 回答"改了 A 之后哪些必须重验"（相邻层影响分析）
- §10.3d 回答"A→E 整条链是不是通的"（跨层完备性检查）
- 两者互补：§10.3 用于日常变更，§10.3d 用于里程碑验证

#### 10.4 下游发现上游缺失或过时的反推机制

下游步骤在设计或实现过程中可能发现上游产出物存在**缺失或过时**（不仅仅是"不一致"）：

| 场景 | 处理方式 |
|------|---------|
| 上游缺失明确（如缺少某权限点） | 反推上游补充定义 → 下游继续 |
| 上游定义过时（如设计中的状态机与代码实现的行为不匹配，但代码行为是正确的） | 启动同步螺旋（§10.3b），以代码为真相源上溯更新上游资产 |
| 归属或粒度存在歧义 | **暂停，提示用户仲裁**——AI 不可替用户做业务判断 |
| 多个缺失/过时点同时存在 | 允许收集后一次性反推，减少来回切换 |

**硬约束：**
- **下游不定义上游资产**——权限点的唯一定义源是 business-workflow
- **缺失必反推**——不可跳过或留空
- **歧义必仲裁**——暂停并提示用户决策

> 此机制以权限控制点为典型场景，但机制本身是通用的——任何下游发现上游缺失均按此流程。

#### 10.5 设计-代码同步与追踪

设计变更确认后应尽快推送到代码。允许多轮设计调整后一次性批量推送。未推送的变更必须在 `design/review/design-code-gap.md` 中记录（`- [ ] {日期}: {变更摘要} — 暂未实现`）。

#### 10.6 收敛判定

当以下两个条件**同时**满足时，视为收敛：
1. 一致性校核连续 2 次未发现新的**实质性问题类型**。不追求零问题，只看是否还在出现新类别。
2. **§8.8 tdd-execute 全量测试通过（0 失败）**——代码行为与设计契约一致。

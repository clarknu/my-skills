# Skill: iterate — 迭代变更编排器

> 功能迭代 + Bug 修复的统一级联引擎。以同步螺旋协议（SFDS §10.3b）为核心编排机制，
> 覆盖从"想加一个功能"到"发现一个 bug"的全部变更场景。

---

## ⛔ 强制两阶段铁律（红线）

> **本 skill 分两阶段执行，阶段一完成后必须停止并等待用户确认，不得自动进入阶段二。**

```
阶段一：分析 + 出计划
    │
    ├── 路径 A：FA1 → FA2 → FA3 → C1（产出：级联修正计划）
    ├── 路径 B：FB1 → FB2 → C1（产出：级联修正计划）
    │
    └── ⛔ 强制停止，等待用户确认。不得自动进入阶段二。
            │
            ├── 用户确认 → 进入阶段二
            ├── 用户修改 → 回到阶段一调整计划
            └── 用户拒绝 → 终止
                    │
阶段二：执行 + 验证
    │
    └── C2（逐步骤执行）→ C3（验证闭环）
```

### 唯一例外：用户明确要求合一步

如果用户在触发本 skill 时**显式声明**以下任一表述，可以跳过阶段一的停止点，分析完成后直接执行：

- "不需要确认，直接改"
- "一步完成"
- "分析完直接执行"
- "不用等我确认"
- "auto" / "yolo" / "--force"

> **默认行为永远是两步走。** 如果用户在阶段一完成、看到计划后说"好的，开始执行"——那是正常的确认流程，不是合一步声明。
> 区分点：**合一步声明必须在触发 iterate 的同一句话里出现。**

## 适用场景

| 场景 | 模式 | 典型触发语 |
|------|------|-----------|
| 加字段、加端点、加页面、改工作流、改业务规则 | **路径 A：功能迭代** | "给订单加一个取消原因字段"、"用户注册流程要加手机验证" |
| 需求变更、原始输入更新 | **路径 A：功能迭代** | "合同附录里说了要支持多设备绑定，设计里还没体现" |
| 设计-实现偏差、代码与设计不一致 | **路径 A 或 B** | 视偏差方向而定 |
| API 500、测试失败、启动崩溃、数据错误 | **路径 B：诊断修复** | "订单接口返回 500"、"3 个测试挂了" |
| 行为不符合预期、权限异常 | **路径 B：诊断修复** | "已取消的订单还能编辑"、"普通用户看到了管理后台" |

## 核心设计理念

### 两条路径，一个引擎

```
用户输入
    │
    ├── "我想加/改 X" ──→ 路径 A：功能迭代
    │                        │
    │                        ├── FA1: 澄清变更意图
    │                        ├── FA2: 定位插入点（哪个 §8.X）
    │                        ├── FA3: 上溯检查（上游需补上下文吗？）
    │                        └── → 进入共享级联引擎 C1
    │
    ├── "X 出问题了" ──→ 路径 B：诊断修复
    │                        │
    │                        ├── FB1: 复现与收集证据
    │                        ├── FB2: 根因追溯（最早偏差步骤）
    │                        └── → 进入共享级联引擎 C1
    │
    └── ⛔ 阶段一结束，强制停止，等待用户确认
              │
              └── 用户确认后：
                       ├── C2: 级联执行（逐步骤委托专业 skill）
                       └── C3: 验证闭环（测试全绿 + Review 通过）
```

### 与同步螺旋协议的关系

本 skill 是 SFDS §10.3b「同步螺旋协议」的**操作化实现**。同步螺旋定义了五步法：

> 标记真相源 → 上溯 → 下传 → 扩散迭代 → 收敛

本 skill 的路径 A 直接映射这五步；路径 B 通过根因追溯自动发现真相源后，汇入同一螺旋。

## 模式分发

### 自动判断

根据用户输入的关键词自动选择路径：

| 用户说… | 判定 | 路径 |
|---------|------|------|
| "加一个…"、"新增…"、"添加…"、"支持…" | 增量变更 | A |
| "改一下…"、"修改…"、"调整…"、"重构…" | 存量修改 | A |
| "需求里说了…但设计/代码没体现" | 需求→设计 gap | A |
| "报错"、"500"、"崩溃"、"挂了"、"不正常"、"bug" | 异常/缺陷 | B |
| "为什么…"（描述意外行为） | 行为不符合预期 | B |
| "测试失败了"、"N 个 red" | 测试失败 | B |

> 如无法自动判定，向用户确认："这是要加新功能/改设计（迭代），还是修问题（排查）？"

## 路径 A：功能迭代

> **适用**：用户主动发起设计或功能的增量/修改。真相源由用户声明，不需要"发现"。

### FA1 — 澄清变更意图

**目标**：把用户的自然语言描述转化为可执行的变更规格。

**执行步骤**：

1. **理解意图**：用户在说什么？要改什么？为什么？
2. **判断范围**：这是单点改动还是连锁改动？
   - 单点："给订单表加一个 `cancelReason` 字段"
   - 连锁："用户注册流程要支持手机验证"（涉及工作流、ER、API、页面）
3. **输出变更摘要**：

```markdown
## 变更意图
- **描述**：{一句话}
- **来源**：用户主动提出 / 原始需求追溯 / 设计审查发现
- **类型**：新增功能 / 修改已有 / 删除 / 重构
- **初步判断影响域**：{涉及哪些领域}
- **初步判断影响层**：§8.1 / §8.2 / §8.3 / §8.3b / §8.4 / §8.5 / §8.6 / §8.7
```

4. **存档**：如果变更是由原始需求驱动的，追加到 `design/01-raw-input/{domain-slug}.md`（标注时间戳 + 变更原因，不修改历史原文）。

### FA2 — 定位插入点

**目标**：确定变更在 SFDS 流水线中从哪一步开始。

**定位方法**：

| 变更内容 | 最早插入步骤 |
|---------|------------|
| 加/改业务规则、状态转换、权限点 | §8.1 工作流 |
| 加/改实体、字段、约束、关系 | §8.2 ER |
| 加/改 API 端点、参数、响应结构 | §8.3 API 设计 |
| 加/改架构策略、缓存、可靠性 | §8.3b 架构设计 |
| 加/改页面、组件、导航 | §8.4 页面设计 |
| 仅改测试用例 | §8.5 TDD |
| 仅改实现代码（不改设计） | §8.6 / §8.7 |
| 需求文档补充（不改下游） | raw-input 存档 |

**输出**：

```markdown
## 插入点
- **最早变更步骤**：§8.X
- **定位依据**：{为什么从这里开始}
```

### FA3 — 上溯检查

> **核心问题**：如果我从 §8.X 开始改，上游有没有缺失的上下文？
> 这个变更依赖的概念/字段/规则，在上游是否已经定义清楚了？

**上溯规则**（逐层向上，每层问一个问题）：

| 当前插入点 | 上溯检查 |
|-----------|---------|
| §8.7 前端代码 | 页面设计有对应页面/组件吗？API 设计有对应端点吗？ |
| §8.6 API 代码 | API 设计有对应端点吗？ER 有所需字段吗？工作流有对应业务规则吗？ |
| §8.5 TDD | API/架构/页面设计是否已经稳定？——通常 TDD 跟着设计走，不独立启动 |
| §8.4 页面设计 | API 有所需端点吗？ER 有页面要展示的字段吗？工作流有对应的用户操作吗？ |
| §8.3 API 设计 | ER 有所需字段和约束吗？工作流有对应的业务操作和权限点吗？ |
| §8.3b 架构设计 | API 设计的非功能需求明确吗？工作流的并发/可靠性要求是什么？ |
| §8.2 ER | 工作流中有对应的业务节点产生/消费这些实体吗？原始需求中有描述吗？ |
| §8.1 工作流 | 原始需求中有对应的功能描述吗？ |

**上溯终止条件**：
- 该层已有完整的定义 → 停止，不需要继续向上
- 该层缺少定义 → **插入点上移**到这一层
- 上溯到 raw-input → 追加原始交互记录

**上溯示例**：

```
用户："给订单列表 API 加一个 cancelReason 字段"
↓
FA2 定位：表面上是 §8.3（API 设计，改响应结构）
↓
FA3 上溯检查：
  §8.3 → §8.2：ER 的 Order 实体有 cancelReason 字段吗？
    ├── 有 → 插入点确认为 §8.3，上溯停止
    └── 没有 → 插入点上移到 §8.2，继续上溯
        §8.2 → §8.1：工作流的"取消订单"节点有产生取消原因吗？
          ├── 有 → 插入点确认为 §8.2
          └── 没有 → 插入点上移到 §8.1
```

**输出**：

```markdown
## 上溯结果
- **最终插入点**：§8.X（可能是调整后的）
- **上溯路径**：{逐层检查记录}
- **上游补充需求**：{哪些层需要补什么}
```

## 路径 B：诊断修复

> **适用**：用户报告问题——行为异常、报错、测试失败等。需要从症状追溯到根因。
> 此路径的核心逻辑从原 `diagnose` skill 移植，精简保留。

### FB1 — 复现与收集证据

1. **理解问题**：预期行为 vs 实际行为
2. **复现**：构造复现步骤，确认可复现
3. **收集证据**：日志、响应体、堆栈、数据库状态、测试结果

**启动服务（如需要）**：从项目 CLAUDE.md 读取启动命令。启动前检查编译和数据库迁移。

### FB2 — 根因追溯

> **核心问题**：这个 bug 的种子是在 SFDS 流水线的哪一步种下的？
> **不是**"哪里改一行代码就好了"，而是"哪个环节最先出现偏差"。

**追溯方法——向上追问法**：

从症状出发，逐层追问："如果上游早就有这条规定/定义，这个 bug 还会发生吗？"

| 回答 | 结论 |
|------|------|
| "不会" → | 根因在上游，继续向上追溯 |
| "会，因为上游已经写清楚了我只是在代码里漏掉了/写错了" → | **根因在当前层，停止** |

**⚠️ 防误判铁律**：代码缺失 ≠ 实现 bug。如果代码没写某字段，但设计文档也没定义该字段 → 根因在设计层，不是代码层。

**症状→步骤分类表**：

| 症状 | 最早偏差步骤 |
|------|------------|
| 功能完全缺失 | 追溯链上第一个"没有"的位置（需求→工作流→ER→API→页面→代码） |
| 业务逻辑错误 | §8.1（工作流缺失规则）或 §8.3（API 设计偏差） |
| API 字段缺失/多余/类型错误 | §8.3（API 设计未定义）或 §8.6（DTO 偏差） |
| 数据库约束/查询报错 | §8.2（ER 缺失约束）或 §8.6（迁移未应用） |
| 页面不存在或入口缺失 | §8.4（页面设计缺失）或 §8.7（前端代码缺失） |
| 测试失败 | §8.5（测试写错）或 §8.6（实现偏差）或 §8.3（设计本身有矛盾） |
| 前后端对接失败 | §8.7（前端调用错）或 §8.6（路由/DTO 不一致） |
| 数据流断裂 | 追溯链上第一个"有产出但下一环节未接收"的位置 |
| 编译失败 | §8.6 或 §8.7（代码编译问题） |

**输出**：

```markdown
## 根因分析
- **最早偏差步骤**：§8.X
- **追溯链**：{从症状到根因的逐层追问记录}
- **分类**：API 相关 / 纯前端
- **根因描述**：{具体是什么偏差——设计遗漏 / 设计错误 / 实现偏差 / 配置问题}
```

## 共享级联引擎 — C1（阶段一的最后一步）

> 两条路径在确定变更起点后，汇入 C1 产出级联计划。

### C1 — 级联计划

**目标**：计算从变更起点到 §8.8→Review 的完整修正链路。

**级联链计算规则**（基于 SFDS §8.0 依赖拓扑）：

| 起点 | 级联链（正向下游） |
|------|------------------|
| raw-input | §8.1 → §8.2 → §8.3 → §8.3b → §8.4 → §8.5 → §8.6+§8.7 → §8.8 → Review |
| §8.1 工作流 | §8.2 → §8.3 → §8.3b → §8.4 → §8.5 → §8.6+§8.7 → §8.8 → Review |
| §8.2 ER | §8.3 → §8.3b → §8.4 → §8.5 → §8.6+§8.7 → §8.8 → Review |
| §8.3 API 设计 | §8.3b → §8.4 → §8.5 → §8.6+§8.7 → §8.8 → Review |
| §8.3b 架构设计 | §8.5 → §8.6 → §8.8 → Review |
| §8.4 页面设计 | §8.5 → §8.7 → §8.8 → Review |
| §8.5 TDD | §8.6+§8.7 → §8.8 → Review |
| §8.6 API 代码 | §8.5（⚠️ 回归用例）→ §8.8 → Review |
| §8.7 前端代码 | §8.8 → Review |

**计划输出格式**：

```markdown
## 级联修正计划

> 变更起点：§8.X | 模式：{功能迭代 / 诊断修复}
> API 相关：{是/否}（如果是，§8.5 必须新增 TDD 回归用例）

| 序号 | 步骤 | 负责 Skill | 变更内容 | 备注 |
|------|------|-----------|---------|------|
| 1 | §8.X | {skill} | {具体要做什么} | 起点 |
| 2 | §8.Y | {skill} | {具体要做什么} | — |
| ... | ... | ... | ... | ... |
| N | §8.8 | tdd-execute | 全量测试 | 含回归用例 |
| N+1 | Review | review | 受影响维度复查 | — |

**并行机会**：§8.6 + §8.7 可并行执行（如两者都在级联链中）
**已检查无需变更**：{列出跳过但已确认的步骤}
```

### ⛔ 阶段一停止点

C1 计划输出后，**必须停止**。输出以下确认提示：

```markdown
---
> ⛔ **阶段一完成。** 请确认以上级联修正计划：
> - 回复 **"确认"** / **"开始"** → 进入阶段二执行
> - 回复修改意见 → 调整计划
> - 回复 **"取消"** → 终止
---
```

> **此停止不可跳过，除非用户在触发 iterate 时已声明合一步模式。**

## ═══════════════════════════════════════
## 阶段二：执行 + 验证
## ═══════════════════════════════════════

> 阶段二仅在用户**明确确认**阶段一的计划后启动。
> 如果用户确认时附带修改意见，先回到阶段一调整计划，再启动阶段二。

### C2 — 级联执行

**执行规则**：

| # | 规则 |
|---|------|
| 1 | **委托不替代**：每一步的实质性修改必须通过对应的专业 skill 执行。本 skill 只做编排和守卫 |
| 2 | **按序执行**：从起点开始，按依赖顺序逐步骤执行。§8.6 和 §8.7 可并行 |
| 3 | **TDD 强制注入**：如果变更属 API 相关范畴（§8.1/§8.2/§8.3/§8.3b/§8.5/§8.6），级联链中的 §8.5 步骤必须新增至少一个针对本次变更的回归测试用例。纯前端变更可豁免 |
| 4 | **每步验证**：每步完成后检查产物更新 + CHANGELOG 记录 |
| 5 | **设计是权威源**：如果代码与设计冲突，以设计为准。代码只能回归设计，不可反向覆盖 |
| 6 | **域注册表同步**：如果变更涉及领域边界调整，先更新 `design/domain-registry.js` |
| 7 | **CHANGELOG 必写**：每步变更后在其负责目录的 `CHANGELOG.md` 中记录 |
| 8 | **零变更也记录**：如果某步骤确实不需要修改，在计划中标注"已检查，无需变更"——不可静默跳过 |

**每步执行模板**：

```markdown
### 🔄 Step X/N：§8.X — {步骤名称}

**负责 Skill**：{skill-name}
**输入**：{已就绪的上游产物}
**动作**：{具体变更内容}
**TDD 回归**：{⚠️ 必须新增 / N/A}

---
{skill 执行输出}
---

**产物验证**：
- [ ] {产物文件} 已更新
- [ ] CHANGELOG 已记录
- [ ] {如果是 §8.5} 回归测试用例已新增

**状态**：✅ / ❌（{原因}）
```

**受阻处理**：如果某步需要用户仲裁（如设计歧义），暂停级联，提交用户决策后从受阻步骤继续。已完成步骤的结果保留。

### C3 — 验证闭环

**三个硬性条件**（必须全部满足）：

1. **§8.8 tdd-execute 全绿**（0 失败，含本次新增回归用例 + 所有已有测试）
2. **Review 无新实质性问题**（或仅受影响维度检查通过）
3. **原始问题/变更意图已确认**：
   - 路径 A：变更已生效，功能符合预期
   - 路径 B：原始 bug 不再复现

**验证报告**：

```markdown
# 迭代报告：{变更简述}

> 日期：{YYYY-MM-DD} | 领域：{domain-slug}
> 模式：功能迭代 / 诊断修复 | 起点：§8.X
> API 相关：是 / 否

## 变更摘要
{一句话}

## 级联执行记录

| 步骤 | 负责 Skill | 变更摘要 | TDD 回归 | 结果 |
|------|-----------|---------|---------|------|
| §8.X | {skill} | {变更} | — | ✅ |
| ... | ... | ... | ... | ... |
| §8.5 | tdd-build | {新增/修改的用例} | ⚠️ | ✅ |
| §8.8 | tdd-execute | 全量测试 ({N} passed) | 含回归 | ✅ |
| Review | review | {受影响维度} | — | ✅ |

## TDD 回归用例（如有）
| 测试方法 | 覆盖场景 | 类型 | 状态 |
```

## 规则与守卫

| # | 规则 | 说明 |
|---|------|------|
| 0 | **强制两阶段** | 阶段一出计划 → ⛔ 停止等确认 → 阶段二执行。除非用户触发时声明合一步 |
| 1 | **变更起点必定位到步骤** | 每次迭代必须明确变更从 SFDS 哪个步骤（§8.X）开始 |
| 2 | **上溯铁律** | 定位插入点后，必须向上游追问：如果上游早就有这条规定，我还需要在这里改吗？答案"不需要"→ 起点上移 |
| 3 | **变更必级联** | 从起点开始，按依赖顺序级联执行全部下游步骤，不可跳过 |
| 4 | **API 相关变更必须追加 TDD 回归** | 起点在 §8.1/§8.2/§8.3/§8.3b/§8.5/§8.6 → §8.5 必须产出至少一个回归用例 |
| 5 | **设计是权威源** | 修正中如代码与设计冲突，以设计为准 |
| 6 | **委托不替代** | 每步实质性修改由对应专业 skill 执行，iterate 只做编排 |
| 7 | **域注册表同步** | 领域边界调整前先更新 `design/domain-registry.js` |
| 8 | **CHANGELOG 追踪** | 每步变更在对应目录的 `CHANGELOG.md` 中记录 |
| 9 | **原始输入存档** | 用户的需求澄清、决策、仲裁追加到 `design/01-raw-input/` |
| 10 | **全绿方可收敛** | 终点是 tdd-execute 全量通过 + Review 无新问题 |
| 11 | **中文输出** | 所有分析、计划、报告用中文 |
| 12 | **修完自动启动服务** | 验证闭环后，默认启动 API 服务（从 CLAUDE.md 读取启动命令） |

## 注意事项

1. **不是所有变更都需要全量级联。** 如果起点在 §8.6（纯代码修正，设计正确），级联只需 §8.6 → §8.5(⚠️ 回归) → §8.8 → Review。

2. **API 相关问题必须追加 TDD 回归（规则 #4）。** 这不是可选项。回归用例的价值在于记录"这个变更曾经发生过，以后不允许回归"。

3. **用户的需求澄清等同于 raw-input 变更。** 迭代过程中用户说"其实应该是这样……"→ 先存档到 `design/01-raw-input/`，再从 §8.1 开始级联。

4. **越早修正成本越低但影响面越大。** §8.1 改一行 → 下游全链重验。§8.6 改一行 → 只需重测。但如果在 §8.6 打补丁掩盖 §8.1 的设计缺陷，下次还会出问题。**本 skill 的价值就是阻止短视的局部修补。**

5. **路径 A 的上溯检查是防漏关键。** 很多迭代问题的根源是"用户说改 API，其实 ER 还没这个字段"——上溯检查就是防止这种"从半空中开始改"的情况。

6. **阶段一 ≠ 空谈。** 阶段一需要实际读文件、查代码、做上溯——不是"猜一个计划"。计划必须基于真实代码和设计现状。

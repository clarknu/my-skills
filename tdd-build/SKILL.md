# TDD Build — 测试设计与编码 Skill

## 适用场景

| 场景 | 说明 |
|------|------|
| **API 设计完成后** | 在写 API 实现代码之前，先设计测试用例、编写测试代码，用测试定义"正确"是什么 |
| **业务流程变更后** | 业务流程改了，对应的 API 行为也要变——先设计/更新测试再改实现 |
| **页面功能新增后** | 页面新增了功能组件，对应的 API 调用场景需要测试覆盖 |
| **设计变更后** | API 设计、ER 或工作流变更后，更新测试设计与代码 |

> **本 skill 只负责测试的设计与编码（Phase 1+2）。测试的执行、报告、失败分析与修复循环由 [`tdd-execute`](../tdd-execute/SKILL.md) 负责。**

---

## 本技能规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | **契约先行** | 测试代码只依赖 API 契约（路径、方法、参数名、响应结构），不依赖任何实现细节（Controller 类名、Service 方法、内部 DTO） |
| 2 | **输入完整** | TDD 设计开始前必须采集全部三项输入：API 设计文档 + ER 数据 + 业务工作流 + 页面功能设计 + 域注册表 |
| 3 | **覆盖度达标** | 正常路径 100% / 参数校验全覆 / 状态转换全覆 / 错误场景 ≥80% / 边界条件 ≥80% / 权限全等级 |
| 4 | **独立可重复** | 每个测试用例独立运行，不依赖其他用例的执行顺序。使用独立测试数据库（通过 DI 容器拦截切换，不动生产配置） |
| 5 | **失败回溯链清晰** | 测试失败时按"测试代码→API实现→API设计→业务流程"逐级回溯定位（由 tdd-execute 执行） |
| 6 | **中文输出** | 所有 TDD 报告、摘要、汇报内容必须使用中文撰写，不得输出英文报告 |

---

## Phase 1：TDD 设计

### 1.1 输入采集

读取以下输入：

| 输入 | 来源 | 提供什么 |
|------|------|---------|
| **API 设计文档** | `design/04-platform-api/domains/{domain-slug}-api.md` | 端点列表、请求参数、响应格式、错误场景、状态机 |
| **ER 数据** | `design/03-entity-relationship/data/{domain-slug}.js` | 实体字段约束、关系定义、枚举值——用于设计字段校验/唯一约束/引用完整性测试 |
| **业务工作流** | `design/02-business-workflow/data/{domain-slug}.js` | 操作步骤顺序、状态转换条件、业务规则约束 |
| **页面功能设计** | `design/{client-slug}/data/tree.js` | 功能组件列表、前端操作流程、页面状态 |
| **后端架构设计（L2+）** | `design/05-backend-architecture/data/*.js` | 架构承诺：幂等端点列表、缓存策略、可靠性策略（超时/重试/熔断/降级）、数据一致性策略（Outbox/Saga）、可观测性要求（健康检查/审计日志/Correlation ID）、安全策略（脱敏/加密）、无状态要求 |
| **域注册表** | `design/domain-registry.js` | 领域 slug、跨域引用 |

### 1.2 测试用例设计方法

对每个功能场景，按以下模板设计测试用例：

```markdown
### TC-{编号}: {测试标题}

- **正向追溯链**: workflow:{node_id}.outputs.{field} → API:{method} {path}.consumes[].{param} → page:{page_id}.sends[].{field}
- **反向追溯链**: page:{page_id}.page_input.params[] ← page:{prev_page}.page_output.params[] ← API:{method} {path}.produces[].{field} ← ER:{entity}.{field}
- **用户旅程**: {journey-name}（来自 business-workflow start→end 路径）

> **注意**：正向/反向追溯链和用户旅程字段在初始设计阶段可选填写，在 §8.9 复审阶段和 review 一致性检查时强制要求。
- **类型**: 正常流程 / 异常流程 / 边界条件 / 状态转换 / 并发 / 幂等 / 缓存 / 故障恢复 / 架构承诺
- **前置条件**: {数据准备、认证状态、业务状态前提}
- **调用序列**:
  1. {请求方法 + 路径 + 关键参数} → 预期响应码
  2. {请求方法 + 路径 + 关键参数} → 预期响应码
- **断言清单**:
  - ✅ {响应字段} = {预期值}
  - ✅ {数据库状态} = {预期状态}
  - ✅ {副作用} = {预期效果}
- **边界条件**:
  - ⚠️ {参数缺失 → 预期 400}
  - ⚠️ {重复请求 → 幂等/409}
  - ⚠️ {越权访问 → 401/403}
```

#### 1.3.0 TDD 分级策略（按复杂度决定测试深度）

> **设计意图：** 不是所有项目都需要全量测试。测试深度应与项目复杂度匹配，
> 避免 L2 项目被迫写 L4 级别的测试。

**复杂度 → 测试维度映射：**

| 测试维度 | L1 简单CRUD | L2 复杂单体 | L3 模块化单体 | L4 分布式 |
|---------|:----------:|:----------:|:----------:|:--------:|
| **对外 API · 正常路径** | ✅ | ✅ | ✅ | ✅ |
| **对外 API · 参数校验** | ✅ | ✅ | ✅ | ✅ |
| **对外 API · 错误场景** | 🟡 关键端点 | ✅ | ✅ | ✅ |
| **状态转换** | — | ✅ | ✅ | ✅ |
| **业务规则** | — | ✅ | ✅ | ✅ |
| **边界条件** | 🟡 | ✅ | ✅ | ✅ |
| **权限** | 🟡 | ✅ | ✅ | ✅ |
| **幂等性** | — | ✅ | ✅ | ✅ |
| **并发冲突** | — | 🟡 | ✅ | ✅ |
| **缓存命中/失效** | — | 🟡 | ✅ | ✅ |
| **外部依赖失败/熔断** | — | — | ✅ | ✅ |
| **事务回滚** | — | 🟡 | ✅ | ✅ |
| **事件发布/Outbox** | — | — | ✅ | ✅ |
| **超时/降级** | — | — | ✅ | ✅ |
| **健康检查** | 🟡 | ✅ | ✅ | ✅ |
| **审计日志** | — | 🟡 | ✅ | ✅ |
| **HTTP 服务间契约** | — | — | — | ✅ |
| **消息 Schema** | — | — | — | ✅ |
| **gRPC/WebSocket** | — | — | — | 🟡 |
| **前端冒烟测试**（静态） | ✅ | ✅ | ✅ | ✅ |
| **前端运行时测试**（§8.9 执行） | — | ✅ 导航+表单+权限 | ✅ + 缓存 | ✅ + 缓存 |
| **前端完整 UI 测试**（像素级） | — | — | 🟡 高风险页面 | 🟡 高风险页面 |

> **图例：** ✅ = 必做，至少 1 个用例 / 🟡 = 建议做，可按需裁剪 / — = 不适用，跳过
>
> **判定规则：** 复杂度等级从 `backend-architecture-design` 的 `system-topology.js` 读取。
L1 项目跳过架构相关测试维度；L2+ 项目逐级解锁更多维度。

> **执行时机：** TDD 的 API 测试和前端测试部分在 §8.3+§8.5 完成后即可启动（不依赖 §8.4）。
> 架构承诺测试（§1.3.1）和契约测试（§1.3.2）在 §8.4 完成后增量补充。
> 这意味着 TDD 设计编码可以与后端架构设计**并行推进**——架构设计完成前先写 API 测试和前端测试，
> 架构设计完成后再补架构承诺测试。

### 1.3 覆盖度要求

以下为 L1 基础覆盖要求（L2+ 项目的扩展要求见 §1.3.1 和 §1.3.2）：

#### 1.3.1 架构承诺测试覆盖（L2+ 项目，详见 §1.3.0 分级表）

> 如果项目复杂度 ≥ L2 且存在后端架构设计资产，以下维度必须覆盖。
> 这些测试验证的不是业务功能是否正确，而是**架构承诺是否兑现**。

| 覆盖维度 | 最低要求 | 来源资产 |
|---------|---------|---------|
| **幂等性** | `idempotency.requiredEndpoints` 中的每个端点至少 1 个重复请求用例（同 Idempotency-Key 两次 POST → 第二个返回 200 而非重复创建） | `resilience-policy.js` |
| **并发冲突** | 涉及状态变更的端点至少 1 个并发用例（两个请求同时修改同一资源 → 一个成功一个 409） | `resilience-policy.js` |
| **缓存命中** | 标注缓存的 GET 端点至少 1 个缓存命中用例（首次 200 → 再次 200 且响应时间显著降低或命中缓存标记） | `caching-strategy.js` |
| **缓存失效** | 标注失效条件的缓存至少 1 个失效后重新获取用例（数据变更后缓存被清除 → 下次 GET 返回新数据） | `caching-strategy.js` |
| **外部依赖失败** | `circuitBreaker.targets` 中的每个外部依赖至少 1 个失败用例（Mock 外部服务超时/500 → 验证熔断/降级行为） | `resilience-policy.js` |
| **事务回滚** | 涉及多表写入的端点至少 1 个部分失败回滚用例（中间步骤抛异常 → 验证前面已写入的数据被回滚） | `data-consistency.js` |
| **事件发布** | 有副作用的端点至少 1 个事件发布验证用例（订单创建后 → 验证 Outbox 表有对应事件记录） | `data-consistency.js` |
| **健康检查** | 1 个健康检查端点用例（GET /api/v1/health → 200 + healthy/degraded，各检查项状态正确） | `observability-policy.js` |
| **审计日志** | `auditLog.events` 中的每个事件至少 1 个审计记录验证用例（触发事件 → 验证审计日志表有对应记录） | `observability-policy.js` |
| **权限绕过** | 每个权限等级至少 1 个越权访问用例（低权限用户访问高权限端点 → 403） | `security-policy.js` |
| **超时/降级** | 标注超时配置的端点至少 1 个超时用例（Mock 慢响应 → 验证超时后返回降级响应） | `resilience-policy.js` |

#### 1.3.2 微服务间契约测试覆盖（L4+ 项目，详见 §1.3.0 分级表）

> 当项目复杂度 ≥ L4（分布式服务），内部模块以独立微服务形式部署，
> 服务间通过 HTTP、消息队列、gRPC、WebSocket 等方式通信。
> 这些通信链路的契约必须独立验证——不能仅依赖对外 API 测试间接覆盖。

| 覆盖维度 | 最低要求 | 来源资产 | 测试方法 |
|---------|---------|---------|---------|
| **HTTP 服务间契约（Consumer 端）** | `module-boundaries.js` 中每个跨服务 HTTP 调用 ≥ 1 个 Consumer 契约用例：Consumer 定义对 Provider 的期望（请求格式 + 预期响应）→ Provider stub 验证能满足 | `module-boundaries.js` | Consumer-Driven Contract (CDC)：Mock Provider，验证 Consumer 发出的请求格式正确 |
| **HTTP 服务间契约（Provider 端）** | 每个暴露 HTTP 接口的服务 ≥ 1 个 Provider 验证用例：验证 Provider 的实际实现满足所有 Consumer 的期望 | `module-boundaries.js` | Provider 端加载所有 Consumer 契约 → 验证实际响应匹配 |
| **消息队列 Schema** | `event-contracts.js` 中每个事件类型 ≥ 1 个 Schema 验证用例：事件名称、字段名、字段类型与 Schema 定义一致 | `event-contracts.js` | Schema Registry 校验 + 序列化/反序列化兼容性测试 |
| **消息队列行为** | 每个事件类型 ≥ 1 个端到端消息用例：发布者发布 → 消费者接收并处理 → 验证 ACK/NACK 行为 + 重试 + 死信 | `event-contracts.js` | 嵌入式 MQ broker（如 Testcontainers）或 Mock |
| **gRPC 契约兼容性** | 每个 Proto service ≥ 1 个向后兼容用例：新增字段不破坏旧 Consumer、删除字段有预留号、枚举值仅追加 | `deployment-profile.js` | Proto lint（buf breaking） + 旧版 Consumer stub 调用新版 Provider |
| **WebSocket 生命周期** | 每个 WebSocket 端点 ≥ 1 个连接生命周期用例：握手 → 心跳 → 消息收发 → 断线重连 → 正常关闭 | `deployment-profile.js` | 测试用 WebSocket 客户端 + 超时/断线模拟 |

**测试目录组织（L4+）：**

```
tests/
├── integration/            ← 对外 API（HTTP 端到端，全服务启动）
│   └── {domain}/
├── contracts/              ← 服务间契约（L4+）
│   ├── consumer/           ← Consumer 期望测试（Mock Provider）
│   │   └── {domain}/
│   ├── provider/           ← Provider 满足测试（Mock Consumer）
│   │   └── {domain}/
│   └── schema/             ← 消息/gRPC Schema 兼容性测试
└── unit/                   ← Service 层单元测试（不走网络）
    └── {domain}/
```

> **契约测试与对外 API 测试在同一流水线步骤执行**（§8.9 tdd-execute），
> 区别仅在于测试代码的目录和 Mock 策略不同。

#### 1.3.3 前端运行时架构测试

> **前置说明：** 前端设计资产与代码之间的**静态一致性比对**（页面→API 映射、数据绑定字段存在性、
> 权限 ID 有效性、表单字段名一致性）由 `mobile-code-gen` / `desktop-code-gen` 的一致性检查模式 +
> `review` 的 D6/D7 维度负责，**不需要 TDD 编写测试代码**——AI 直接扫描 tree.js + 代码文件做比对即可。
>
> 本节只覆盖**需要前端代码实际运行起来**才能验证的测试维度。

| 测试维度 | 数据来源 | 测试方法 | 用例设计 | 用例执行 | 适用等级 |
|---------|---------|---------|---------|---------|---------|
| **导航跳转完整性** | `processes.js` 中的页面跳转路径 | 模拟用户操作流程：起点页面 → 触发跳转 → 验证到达终点页面，URL/路由参数正确 | §8.6（§8.5 完成后） | §8.9（§8.8 完成后） | L2+ |
| **表单提交流程** | `tree.js` 中表单组件 + API 端点 | 填写表单 → 提交 → 验证 API 被调用（参数名+类型+必填校验）、验证成功/失败 UI 反馈 | §8.6（§8.5 完成后） | §8.9（§8.8 完成后） | L2+ |
| **权限控制生效** | `tree.js` 中 `perm_ref` + 工作流 `permissions` | 无权限用户访问受限页面/组件 → 验证被拦截（跳转登录或展示无权限提示） | §8.6（§8.5 完成后） | §8.9（§8.8 完成后） | L2+ |
| **前端缓存行为**（如有前端架构设计） | 前端架构的 CDN/TTL 配置 | 首次加载 → 验证 200；二次加载 → 验证 304 或 Cache-Control 命中 | §8.6（§8.5 完成后） | §8.9（§8.8 完成后） | L3+ |

> **关键原则：**
> - **设计时（§8.6）只设计测试用例**——不写测试代码。用例设计只需要 `tree.js` + `processes.js`，不依赖前端代码。
> - **执行时（§8.9）才写并运行测试代码**——此时 §8.8 前端代码已完成，可以在真实页面上执行。
> - **静态比对不需要 TDD**——页面设计↔代码的对齐检查由 code-gen skill 一致性检查模式负责，
>   用 AI 扫描比对即可，不需要写可执行的测试代码。

**前端运行时测试目录：**

```
tests/
├── frontend/               ← 前端运行时架构测试（§8.9 执行）
│   └── {client-slug}/
│       ├── test_navigation.js       ← 导航跳转完整性
│       ├── test_form_submit.js      ← 表单提交流程
│       ├── test_permission_ui.js    ← 权限控制生效
│       └── test_cache_behavior.js   ← 前端缓存行为（L3+）
```

## Phase 2：TDD 编码

### 2.1 编写测试代码

基于 Phase 1 的设计文档，编写实际的测试代码。

**测试代码原则：**

| 原则 | 说明 |
|------|------|
| **一个用例一个方法** | 每个 TC-XXX 对应一个测试方法，方法名体现测试场景 |
| **测试数据工厂** | 通用的测试数据准备抽取为工厂方法，避免重复 |
| **Mock 外部依赖** | 第三方服务、消息队列等外部依赖使用 Mock |
| **数据库隔离（DI 拦截）** | 见下方 §2.1.1「测试数据库隔离规则」 |
| **可重复执行** | 每个测试用例独立，不依赖其他用例的执行顺序 |

### 2.1.1 测试数据库隔离规则 —— DI 容器拦截模式

**核心原则：测试代码通过 DI 容器替换来切换数据库，永远不通过修改配置文件或环境变量来改变数据库行为。**

生产环境的数据库连接串写在配置文件中，由应用入口读取。
测试基础设施通过框架提供的 DI 拦截机制，在容器构建后将生产注册**移除**并**替换**为测试专用实例。
配置文件和入口代码对"测试模式"零感知——没有 `if (isTest)` 分支，没有 `TestingConnectionString` 配置键。

> **为什么是 DI 拦截而不是配置文件切换？**
> - 配置文件切换意味着入口代码需要感知"当前是测试还是生产"，污染入口代码
> - 环境变量切换（如 `APP_ENV=Testing`）是间接耦合——改了变量却期望行为变化，出问题时难以追踪
> - DI 拦截是显式的、可读的、集中在测试基础设施中的——TestHarness 只需"移除生产注册 → 注册测试替身"两步就把整个依赖替换了
> - 生产代码不携带任何测试负担，测试代码不依赖任何配置文件 magic string

**适用范围：** 本规则不仅限于数据库。所有需要在测试中隔离的外部依赖（数据库、消息队列、对象存储、缓存服务、第三方 API 客户端等），统一通过 DI 容器拦截替换。

**数据库无关声明：** 本规则与数据库类型无关。无论是关系型（SQLite / PostgreSQL / MySQL / SQL Server）、文档型（MongoDB）、或其他存储系统，隔离策略完全相同——生产环境通过配置连接真实实例，测试环境通过 DI 拦截替换为内存实例/测试容器/轻量替代。

#### 通用模式

```
生产入口（app entry: main.ts / App.java / Program.cs）
  │
  └── 读取配置 → 注册真实依赖（DB / MQ / Storage / Cache）
      │
      └── 不感知测试/生产模式差异
          │
          ▼
测试基础设施（TestHarness / TestFactory / TestModule）
  │
  └── 继承/包装生产入口
      │
      ├── 1. 定位 DI 容器中的生产注册
      ├── 2. 移除生产注册
      ├── 3. 注册测试替身     ← 内存DB / Mock / Stub
      └── 4. 其他测试专用替换（后台服务禁用、外部服务 Stub 化）
```

#### 框架对应关系

| 框架 | 生产入口 | DI 拦截入口 | 替换 API |
|------|---------|------------|---------|
| ASP.NET Core | 入口文件 + 配置文件 | `WebApplicationFactory<TProgram>` → `ConfigureServices` | `Remove(生产注册)` → `Add(测试替身)` |
| Spring Boot | `Application.java` + `application.yml` | `@TestConfiguration` + `@Primary` Bean 覆盖 | `@Bean` 覆盖或 `@Profile("test")` |
| NestJS | `main.ts` + `AppModule` | `Test.createTestingModule()` → `overrideProvider()` | `overrideProvider().useValue()` |
| Express / Fastify | `app.ts` + 配置 | 测试文件直接构造 app 实例并注入 Stub | 手动替换中间件/服务实例 |
| Django | `settings.py` | `@override_settings` + `TestCase` 的 `setUp` | `override_settings(DATABASES={...})` |

#### 通用模式总结

> **本质就是 3 步：** (1) 定位 DI 容器中的生产注册 → (2) 移除它 → (3) 注册测试替身。
> 生产入口文件对测试模式零感知。具体替换 API 由各框架的 DI 机制决定，与数据库类型无关。

#### 测试替身选择指南

| 替身类型 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| **内存数据库**（SQLite :memory: / H2 / Embedded MongoDB） | 开发阶段、快速迭代、CI 轻量运行 | 零外部依赖、启动极快、隔离天然 | 方言差异可能导致生产 Bug 漏过 |
| **测试容器**（Testcontainers: PostgreSQL / MySQL / MongoDB） | 接近上线、对 SQL 方言敏感的查询 | 与生产环境完全一致 | 需要 Docker、启动较慢 |
| **Mock / Stub** | 单元测试、不涉及真实查询的 Service 层测试 | 极快、精确控制返回值 | 不验证实际查询逻辑 |

> **推荐策略：** 日常开发用内存数据库（快速迭代），CI 流水线中增加一个 Testcontainers 阶段（方言验证）。两者都通过 DI 拦截切换，不需要改任何生产代码。

**文件组织：**

```
tests/
├── integration/            ← 对外 API（HTTP 端到端，全服务启动）
│   └── {domain}/
│       ├── test_{feature1}.cs
│       ├── test_{feature2}.cs
│       └── conftest.cs      # 共享 fixture/test data
├── contracts/              ← 服务间契约（L4+）
│   ├── consumer/
│   │   └── {domain}/
│   ├── provider/
│   │   └── {domain}/
│   └── schema/
└── unit/                   ← Service 层单元测试（不走网络）
    └── {domain}/
```

### 2.2 与 API 代码的关系

**测试代码完全独立于 API 实现**。测试只依赖 API 设计文档中的接口契约：
- 路径、方法、参数名、参数类型
- 响应结构、状态码
- 错误格式

测试代码不依赖任何 API 实现细节（Controller 类名、Service 方法名等）。

## 一致性检查模式

本模式检查 TDD 测试是否完整覆盖了 API 设计文档中定义的端点，以及测试代码是否与 TDD 设计文档一致。

### 触发场景

| 场景 | 说明 |
|------|------|
| **API 设计变更后** | 新增/修改端点后，检查 TDD 测试是否同步更新 |
| **API 代码实现后** | 实现完成后，检查测试是否覆盖了所有必要场景 |
| **常规复查** | review skill 在 TDD 闭环一致性维度中委托调用 |

### 输入规格

| 输入 | 来源 |
|------|------|
| API 设计文档 | `design/04-platform-api/domains/{slug}-api.md` |
| TDD 设计文档 | `design/07-tdd/{slug}-tdd-design.md` |
| 测试代码 | `tests/` 目录下的测试文件 |

### 检查步骤

1. **设计→代码用例覆盖检查**：TDD 设计文档中的每个 TC-XXX，对照测试代码中的测试方法，标记设计有但代码缺失的用例
2. **API 端点覆盖检查**：API 设计文档中的每个端点，对照测试代码中测试的方法和路径，标记缺少测试的端点
3. **断言一致性检查**：TDD 设计文档中每个 TC 的断言清单，对照测试代码中的断言语句，标记不一致的断言
4. **编译通过检查**：确认测试代码可编译，标记编译错误
5. **追溯完整性检查**：每个测试用例是否有正向/反向追溯链标注。缺失 → issue type=`untraced_tc`。追溯链在某一层断裂 → issue type=`trace_chain_broken`。测试用例无法反推回任何 workflow 节点或原始需求 → issue type=`orphan_tc`。

### 输出格式

> 输出格式遵循共享规范：`.claude/skills/_shared/consistency-check-format.md`。
> 本 skill 的 `type` 枚举：`missing_tc`, `missing_endpoint_test`, `assertion_mismatch`, `compile_error`, `architecture_contract_missing`。
> 本 skill 的 `source` 枚举：`api-to-tdd`, `design-to-code`。

```json
{
  "summary": {
    "domain": "domain-slug",
    "tc_in_design": 0,
    "tc_in_code": 0,
    "matched_tc": 0,
    "total_issues": 0,
    "code_compiles": true
  },
  "issues": [
    {
      "severity": "high",
      "type": "missing_endpoint_test",
      "detail": "API 端点 GET /api/v1/competitions 在测试代码中无对应测试方法",
      "suggestion": "添加 test_get_competitions_list 方法"
    }
  ]
}
```

### 使用方式

1. **作为独立检查**：对 Claude 说"使用 tdd-build 检查 {domain-slug} 的测试覆盖一致性"
2. **被 review 调用**：review skill 在 TDD 闭环一致性维度（D8）中委托本技能执行静态覆盖检查

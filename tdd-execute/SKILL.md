# TDD Execute — 测试执行与验证 Skill

## 适用场景

| 场景 | 说明 |
|------|------|
| **流水线 §8.8** | API 代码 + 前端代码全部实现完成后，执行全量测试验证 |
| **TDD 迭代闭环（标准 §10.3c）** | 设计变更后，TDD 更新 → 执行验证 → 失败分类 → 修复循环的完整闭环 |
| **Review 修复循环** | Review 每轮修复完成后，重新执行测试确认修复未引入新问题 |
| **提交前守卫** | 任何代码变更提交前，执行全量测试确保全部通过 |
| **回归验证** | 大规模修改后确认已有功能未被破坏 |
| **CI/CD 集成** | 作为自动化质量门禁的一部分 |

> **本 skill 只负责测试的执行、报告、失败分析与修复循环（Phase 3）。测试的设计与编码由 [`tdd-build`](../tdd-build/SKILL.md) 负责。**

---

## 前置条件（强制）

本 skill 必须在以下产物全部就绪后才能启动：

| 产物 | 来源 | 就绪信号 |
|------|------|---------|
| **测试代码** | `tdd-build` Phase 2 输出 | `tests/` 目录下存在测试项目，编译通过（0 错误） |
| **API 实现代码** | 流水线 §8.6 | `src/server/` 下 Controller/Service/DTO 已实现，编译通过 |
| **ORM 模型** | 流水线 §8.6 | EF Core 迁移已生成，数据库已同步 |
| **数据库** | 流水线 §8.6 → `dotnet ef database update` | 实际数据库 schema 与 EF model snapshot 一致 |
| **前端代码** | 流水线 §8.7 | `src/` 下前端代码已实现（非必需，但会影响集成测试） |

### 启动前置检查清单

- [ ] 测试项目编译通过（`dotnet build` 测试项目，0 错误）
- [ ] API 项目编译通过（`dotnet build` 服务端项目，0 错误）
- [ ] 数据库已迁移到最新（`dotnet ef database update` 无错误）
- [ ] 无待执行的 EF 迁移（`dotnet ef migrations list` 全部 Applied）

**任一前置条件未满足，tdd-execute 不得启动。应报告缺失项并等待补充。**

## 本技能规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | **实际执行铁律** | 必须亲自执行 `dotnet test`，记录实际输出。不得仅凭 TDD 报告中"全部通过"的声明跳过实际执行 |
| 2 | **全项目覆盖** | 运行所有测试项目（`.csproj`），不得遗漏任何一个 |
| 3 | **失败必溯源** | 每个失败测试按"测试代码→API实现→API设计→业务流程"逐级回溯，定位根因 |
| 4 | **修复必重跑** | 任何修复（测试代码/API实现/API设计）后，必须重新执行全量测试 |
| 5 | **中文输出** | 所有报告、摘要、分析必须使用中文撰写 |
| 6 | **报告随测试代码归档** | 执行报告放在测试工程的 `test-reports/` 目录下（技术栈无关） |

## Step 1：执行全量测试

### 1.1 发现所有测试项目

首先列出所有测试项目（`.csproj` 文件中的测试项目）：

```bash
# 搜索所有测试项目
find tests/ -name "*.csproj" | sort
```

**铁律：不得遗漏任何测试项目。** 如果存在 N 个测试项目，必须全部运行 N 次。

### 1.2 逐项目运行

对每个测试项目执行：

```bash
dotnet test <test-project>.csproj --verbosity normal
```

> 不使用 `--no-build` —— 每次都从编译开始，确保测试的是当前最新代码。

### 1.3 记录原始输出

每个测试项目的原始输出（包括通过/失败/跳过数、失败堆栈信息）必须完整保留，
作为后续分析的基础证据。

## Step 2：失败分类与溯源

### 2.1 结果分类表

| 测试结果 | 含义 | 处理方式 |
|---------|------|---------|
| ✅ 通过 | API 实现符合设计契约 | 无需处理 |
| ❌ 测试代码问题 | 测试用例写错了（断言不对、参数不对、mock 配置错误） | 修正测试代码 → 重新运行 |
| ❌ API 实现问题 | 测试正确但 API 行为不符合设计 | 修正 API 实现 → 重新运行 |
| ❌ API 设计问题 | 测试和实现都对不上设计文档 | 回溯到 §8.3，修正 API 设计 → 更新测试 → 重新运行 |
| ❌ 业务流程问题 | 测试反映的业务逻辑与工作流不一致 | 回溯到 §8.1，修正业务工作流 → 更新设计 → 更新测试 → 重新运行 |
| ⏭️ 跳过 | 测试被显式跳过（`[Fact(Skip="...")]`） | 审查跳过理由是否仍然有效 |
| ❌ 追溯链断裂 | 测试正确、实现正确，但数据字段在跨层传递时丢失 | 回溯到对应设计层修正 → 更新测试 → 重新运行 |

### 2.2 溯源方法

对每个失败用例，按以下顺序逐级回溯：

```
测试失败
    │
    ├── ① 检查断言是否正确（对照 API 设计文档中的响应结构）
    │     └── 断言错误 → 问题在测试代码 → 修正测试代码
    │
    ├── ② 检查 API 实现是否与 API 设计一致（路由、参数、响应码）
    │     └── 实现偏差 → 问题在 API 代码 → 修正 API 实现
    │
    ├── ③ 检查 API 设计是否与业务工作流一致（状态转换、业务规则）
    │     └── 设计不一致 → 问题在 API 设计 → 回溯 §8.3
    │
    └── ④ 检查业务工作流是否与原始需求一致
          └── 工作流缺失 → 问题在业务流程 → 回溯 §8.1
    │
    ├── ⑤ 检查追溯链是否完整
    │     正向：workflow.outputs → API.consumes → page.sends → code
    │     反向：page.page_input ← page.page_output ← API.produces ← ER.field
    │     └── 追溯链断裂 → 回溯到对应设计层
```

### 2.3 批量分类输出

```markdown
## 失败分类

### 测试代码问题（N 个）
| 测试方法 | 失败原因 | 修复方案 |
|---------|---------|---------|
| test_xxx | 断言期望 200 但 API 设计定义为 201 | 修正断言为 201 |

### API 实现问题（N 个）
| 测试方法 | 失败原因 | 修复方案 |
|---------|---------|---------|
| test_yyy | 响应体缺少 `createdAt` 字段 | 在 DTO 中添加 `createdAt` |

### API 设计问题（N 个）
| 测试方法 | 失败原因 | 修复方案 |
|---------|---------|---------|
| test_zzz | 状态转换 `pending→approved` 未定义 | 回溯 §8.3 补充 |

### 业务流程问题（N 个）
| 测试方法 | 失败原因 | 修复方案 |
|---------|---------|---------|
| test_www | 工作流缺少退款状态转换 | 回溯 §8.1 补充 |
```

## Step 3：修复循环

### 3.1 循环规则

```
┌──────────────────────────────────────────┐
│                                          │
│  ① 运行全量测试                           │
│       │                                   │
│       ├── 全部通过 → 退出循环，进入 Step 4 │
│       │                                   │
│       └── 有失败 → ② 分类溯源              │
│                     │                     │
│                     ↓                     │
│              ③ 按分类修正                  │
│              · 测试代码问题：修正测试       │
│              · API 实现问题：修正实现       │
│              · API 设计问题：回溯 §8.3     │
│              · 业务流程问题：回溯 §8.1     │
│                     │                     │
│                     ↓                     │
│              ④ 重新运行全量测试             │
│                     │                     │
│                     └── 回到 ① ──────────→ │
│                                          │
└──────────────────────────────────────────┘
```

### 3.2 修正约束

| 修正类型 | 允许的动作 | 禁止的动作 |
|---------|----------|----------|
| **测试代码修正** | 修正断言、修正参数、修正 mock 配置 | 降低覆盖标准、删除"难测"的用例 |
| **API 实现修正** | 修正 Controller/DTO/Service 逻辑 | **修改 API 设计契约**（路由/参数定义/响应结构） |
| **API 设计修正** | 通过 §8.3 api-design skill 修改设计文档，同步更新测试 | 只改测试不改设计文档 |
| **业务流程修正** | 通过 §8.1 business-workflow skill 修改工作流 | 跳过设计层直接改代码 |

> **铁律：** 如果修复需要改动 API 设计或业务流程，必须通过对应的设计 skill 执行，
> 不可直接在修复循环中修改设计文件。这是"代码不能主动变更业务逻辑"原则的体现（标准 §10.2）。

### 3.3 修正记录

每次修复循环中的每个修正，记录：

```markdown
| 循环轮次 | 失败用例 | 分类 | 根因 | 修正动作 | 修正文件 | 修正结果 |
|---------|---------|------|------|---------|---------|---------|
| R1 | test_xxx | API 实现问题 | Controller 未返回 201 | 在 Create 方法添加 CreatedAtAction | Controller.cs | ✅ |
| R1 | test_yyy | 测试代码问题 | 断言写错 | 修正 StatusCode 期望值 | Test.cs | ✅ |
| R2 | （无失败） | — | — | — | — | ✅ 全绿 |
```

## Step 4：输出报告

### 4.1 报告格式

测试全部通过后，输出测试报告。**报告必须放在测试代码目录下**（与测试项目的 `.csproj` 文件同级），属于测试产出物：

```markdown
# TDD 执行报告：{领域名}

> 日期：{YYYY-MM-DD} | 测试项目：{test-project-name} | 执行者：tdd-execute skill

## 整体结果

| 指标 | 数值 |
|------|------|
| 总用例数 | N |
| 通过 | N |
| 失败 | N |
| 忽略 | N |
| 耗时 | Ns |

### 测试项目明细

| 测试项目 | 总用例 | 通过 | 失败 | 跳过 | 耗时 |
|---------|--------|------|------|------|------|
| tests/Boxing.Tests.csproj | N | N | 0 | N | Ns |

### 覆盖统计

| 维度 | 覆盖 | 结论 |
|------|------|------|
| API 端点覆盖 | N/N (100%) | ✅ |
| 状态转换覆盖 | N/N (100%) | ✅ |
| 错误场景覆盖 | N/N (≥80%) | ✅ |
| 边界条件覆盖 | N/N (≥80%) | ✅ |
| 权限覆盖 | N/N (100%) | ✅ |

### 追溯覆盖统计
| 维度 | 覆盖 | 结论 |
|------|------|------|
| 正向链覆盖（workflow→API→page） | N/N | ✅/⚠️ |
| 反向链覆盖（page←API←ER） | N/N | ✅/⚠️ |
| 用户旅程覆盖 | N/N 旅程全覆盖 | ✅/⚠️ |
| 孤儿测试用例（无追溯） | N | ✅/⚠️ |

### 烟雾验证结果
| 旅程 | 状态 | 断点说明 |
|------|------|---------|
| 旅程1：{名称} | ✅ 全通 / 🔴 断点 | {如有断点，说明断裂位置} |

### 修复记录（如有）

| 轮次 | 失败数 | 修正数 | 结果 |
|------|--------|--------|------|
| R1 | 3 | 3 | 3 通过 |
| R2 | 0 | 0 | 全绿 ✅ |

### 健康度评估

| 指标 | 状态 |
|------|------|
| 全部通过 | ✅ / ❌ |
| 覆盖率 ≥ 60% | ✅ / ❌ |
| 无跳过用例 | ✅ / ⚠️ |
| 修复循环 ≤ 3 轮 | ✅ / ⚠️ |
```

**报告归档路径：** `{test-project-root}/test-reports/tdd-execute-report-{date}.md`

> `{test-project-root}` 是测试工程的根目录（无论技术栈——`.NET`、`Python`、`Node.js`、`Java` 等均适用）。
> 在测试根目录下新建 `test-reports/` 文件夹，所有自动化测试报告统一放入该目录。
>
> 示例：
> - .NET：`tests/test-reports/tdd-execute-report-2026-06-20.md`
> - Python：`tests/test-reports/tdd-execute-report-2026-06-20.md`

### 4.2 "全部通过"的定义

以下状态视为"未全部通过"，必须继续修复循环：

| 情况 | 判定 |
|------|------|
| 任何测试项目有失败用例 | ❌ 未通过 |
| 任何测试项目编译失败 | ❌ 未通过 |
| 任何测试项目无法运行（runtime error） | ❌ 未通过 |
| 覆盖度不满足标准（< 60%） | ❌ 未通过（需补充测试 → 回到 tdd-build） |

## Step 5：关键旅程冒烟验证（§8.9 集成）

> **触发条件：** Step 4 全部通过后自动进入。不可跳过。

### 5.1 旅程提取
从域注册表 + business-workflow 中提取 2-3 条最高优先级用户旅程。

### 5.2 逐旅程走查
对每条旅程执行 development-standard §8.9 定义的检查：
- 页面文件齐全
- API 路径可调通（可复用 Step 1-4 已通过的测试结果加速）
- 页面间数据传递完整（page_output.params 全部出现在下游 page_input.params）
- 组件-API 绑定正确（sends 字段匹配 API consumes）
- 旅程终点数据可反推

### 5.3 输出
烟雾报告追加到测试报告末尾，每条旅程 1 行状态（✅ 全通 / 🔴 断点）。

## 6. 被外部过程调用的规范

tdd-execute 被以下上游过程独立调用。每次调用都是完整的 Step 1→5 闭环。

### 6.1 被流水线 §8.8 调用

**触发时机：** §8.6（API 代码）+ §8.7（前端代码）全部完成后。

**调用者：** development-standard 流水线编排。

**期望输出：** 全绿 + 测试报告。

### 6.2 被 Review 修复循环调用

**触发时机：** Review skill §5.2 Step 3（级联检查）。

**调用者：** review skill，在每轮修复的所有 `✅ fix` 问题处理完毕后。

**期望输出：** 确认修复未引入新回归 + 更新测试报告。

**特殊要求：** 如果执行中发现新的失败（上轮通过的测试本轮失败），生成新问题条目反推给 review。

### 6.3 被提交前守卫调用

**触发时机：** 任何代码变更提交前（标准 §10.3 硬约束 #1/#2/#3）。

**调用者：** 开发者或自动化 hook。

**期望输出：** 全绿 → 允许提交；有失败 → 阻止提交。

## 7. 与流水线其他步骤的关系

| 步骤 | 关系 |
|------|------|
| **tdd-build §8.5** | **上游依赖**。tdd-execute 运行的测试代码由 tdd-build 产出 |
| development-standard §8.6（API 实现） | **上游依赖**。API 实现是测试的验证对象，必须在 tdd-execute 之前完成 |
| development-standard §8.7（前端实现） | **上游依赖**。前端代码影响集成测试，应在 tdd-execute 之前完成 |
| development-standard §8.9（e2e/冒烟） | **下游步骤 / 已集成**。§8.9 冒烟验证已集成为本 skill 的 Step 5，全绿后自动执行 |
| development-standard §9（复查门控） | **互相引用**。Review 引用 tdd-execute 的执行报告；tdd-execute 的失败可能触发 review |
| development-standard §10.3（变更传播） | **强制守卫**。代码变更 → 触发 tdd-execute → 全绿方可提交 |

> **关键关系：** tdd-execute 是 tdd-build 的**运行时验证层**。tdd-build 证明"测试写对了设计"，
> tdd-execute 证明"代码实现了设计"。两者合在一起构成完整的红-绿-重构闭环。
> **tdd-execute 不得在 API 代码实现前执行**——此时测试必定失败（API 不存在），
> 这些失败没有分析价值，只是噪音。

## 8. 完成检查清单

| 检查项 | 要求 |
|--------|------|
| 所有测试项目均已运行 | 无遗漏的 `.csproj` |
| 全部测试通过（0 失败） | 每个测试项目 `Failed: 0` |
| 覆盖度达标 | 所有生产项目覆盖率 ≥ 60% |
| 失败已分类溯源 | 每个失败有分类（测试问题/实现问题/设计问题/流程问题） |
| 修复循环记录完整 | 每轮修正有记录（轮次/用例/根因/修正内容/结果） |
| 测试报告已输出 | `test-reports/tdd-execute-report-{date}.md` 存在，中文撰写 |

# Consolidate Raw Input — 原始输入整理 Skill

## 概述

在遵循"设计资产驱动"方法论的项目中，通常会有一个**原始输入存档目录**——
所有设计讨论、需求对话、决策记录都以原始形态追加到此目录。

随着项目演进，该目录中会积累：

- ✅ 最终确认的设计决策
- ❌ 已被推翻的早期方案
- 🔄 多轮迭代中的中间态讨论
- 📐 方法论/参考文档（不应归档）
- 🎵 多媒体附件（录音、文档）

**核心问题：** 所有内容（对的、错的、废弃的）混在一起，后续复查和 AI 处理时会被冗余/失效信息干扰。

**本技能目标：** 将原始输入提炼为**仅包含最新结论**的规范文档，原始文件归档至 `original-logs/` 子目录。

---

## 适用场景

| 场景 | 说明 |
|------|------|
| 积累了大量原始讨论后 | 手动触发整理，清理冗余 |
| 设计方向发生重大变化 | 需要重新梳理"当前到底是怎么定的" |
| 开始新一轮全量复查前 | 先整理原始输入，避免废弃信息干扰复查 |
| 项目阶段性归档 | 整理后提交一个干净的原始输入快照 |

**不属于自动化流水线。** 不会在任何设计/实现/验证类技能中自动触发。
必须用户明确调用（说触发词）才执行。

---

## 前置步骤：定位原始输入目录

不同项目的原始输入目录路径和命名可能不同。执行前先确定目标目录：

1. 优先读取项目中的 `CLAUDE.md` 或 `development-standard`，查找原始输入目录的定义
2. 常见命名：`design/01-raw-input/`、`design/raw-input/`、`docs/raw-input/`
3. 如无法从项目文档中确定 → 询问用户："原始输入文件在哪个目录下？"
4. 如果目录不存在 → 告知用户并停止

> **本技能以下内容用 `{raw-input-dir}` 指代该目录。**

## 处理流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 1. 扫描   │───→│ 2. 分类   │───→│ 3. 分析   │───→│ 4. 规划   │───→│ 5. 确认   │
│ 清点文件   │    │ 判定类型   │    │ 提取结论   │    │ 输出方案   │    │ 用户审核   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └─────┬────┘
                                                                       │
                                                              ┌────────┘
                                                              ▼
                                          ┌──────────┐    ┌──────────┐
                                          │ 6. 写入   │←───│ 5b. 修改  │
                                          │ 生成文档   │    │ 按反馈调  │
                                          └─────┬────┘    └──────────┘
                                                ▼
                                          ┌──────────┐
                                          │ 7. 清旧   │（如 original-logs/ 非空）
                                          │ 清理上轮   │
                                          └─────┬────┘
                                                ▼
                                          ┌──────────┐
                                          │ 8. 归档   │
                                          │ 移动原文件 │
                                          └──────────┘
```

### 步骤 1：扫描（Inventory）

列出 `{raw-input-dir}` 下所有文件（排除 `original-logs/` 子目录和已有整合文档），记录：
- 文件名
- 文件大小（行数）
- 创建/修改日期（从 git history 获取）

### 步骤 2：分类（Classify）

将每个文件归入以下类别：

| 类别 | 判定标准 | 处理方式 |
|------|---------|---------|
| **设计讨论** | 含多轮对话、✅/❓标记、迭代过程 | 提取最终结论，归档原始文件 |
| **最终决策** | 标题含"决策"/"确认"/"最终"，内容无废弃标记 | 内容直接合并到整合文档，归档原始文件 |
| **补充说明** | 对某个领域/主题设计的补充、gap-filling | 合并到对应主题的整合文档，归档原始文件 |
| **方法论/参考** | 定义工作方法而非业务决策 | **不归档**，保留在原位置 |
| **多媒体附件** | .wav / .mp3 / .docx / 图片等非 Markdown 文件 | 直接归档 |
| **已过时标记** | 文件内容明确标注"已废弃"/"superseded"/删除线且整篇无有效内容 | 直接归档，不提取内容 |

**分类原则：**
- 如果文件同时包含方法论内容和业务决策 → 按**主要用途**分类
- 如果无法确定 → 归入"设计讨论"（保守处理，至少会被 AI 分析）

### 步骤 3：分析（Analyze）

对每个需处理的文件，逐文件分析。

#### 3.1 标记最新结论

识别并提取**当前仍然有效**的内容：

| 信号 | 含义 |
|------|------|
| `✅ 确认` / `已确认` / `最终方案` | 当前采用的设计 |
| `已执行` / `已实现` / `代码已同步` | 已在代码中落地 |
| 日期 + `推翻` / `→ 改为` | 后面跟着的是最新结论 |
| `~~删除线~~` / `已删除` / `不需要` | 已废弃，跳过 |
| `❓ 待定` / `待确认` / `TODO` | 仍未决定的开放问题（整合时需特别标注） |

#### 3.2 识别废弃内容

以下内容**不提取**到整合文档：

- 被后续讨论明确推翻的方案
- 已删除的实体/功能/端点
- 早期多方案比较中落选的方案
- 用 `~~删除线~~` 标记的内容
- 纯过程性讨论（"我们考虑了 X，但决定用 Y"——只提取 Y）

#### 3.3 识别跨文件关联

当多个文件讨论同一主题时（例如三个文件都在讨论"权限模型"），标记关联关系。
将关联文件归为一组，后续合并到同一整合文档。

### 步骤 4：规划（Plan）

基于分析结果，产出一份**整合方案**供用户确认：

```markdown
# 原始输入整合方案

## 扫描结果
- 目录：`{raw-input-dir}`
- 文件总数：N 个
- 将保留（不归档）：X 个（方法论/参考文件）
- 将归档：Y 个
- 将生成整合文档：Z 个

## 将保留的文件（不归档）
- {methodology-file-1}.md — 方法论参考，不属于业务讨论
- ...

## 将归档的原始文件
[列出所有将移入 original-logs/ 的文件]

## 将生成的整合文档

### 1. `{topic-slug-1}.md`
**来源文件：** {file-a}.md（相关部分）、{file-b}.md
**涵盖主题：** {主题的简短描述}

### 2. `{topic-slug-2}.md`
**来源文件：** ...
**涵盖主题：** ...

...
```

**整合粒度原则：**
- 一个整合文档覆盖一个内聚的业务主题
- 全局/跨主题的决策放入 `00-global-design-decisions.md`
- 不要过度拆分（一个主题一个文件）也不要过度合并（所有内容塞一个文件）
- 如果拿不准拆分方式，在方案中提出 2-3 种备选让用户选择

### 步骤 5：确认（Confirm）

**必须**将整合方案呈现给用户，等待明确确认（"可以" / "执行" / "确认"）后才进入步骤 6。

- 用户提出修改意见 → 调整方案 → 再次确认
- 用户取消 → 停止流程，不做任何修改

### 步骤 6：写入（Write）

按确认的方案，逐文件生成整合文档。

#### 6.1 文档结构模板

```markdown
# {主题名称}

> 文档 ID: REQ-{topic-slug} | 最后更新：YYYY-MM-DD
> 最后更新：YYYY-MM-DD | 从 {N} 个原始讨论文件整合
> 本文档为整合后的当前设计结论。原始讨论记录见 `original-logs/`。

## 一、核心决策

[该主题下最重要的 3-5 个顶层决策，简洁陈述]

> **编号规则：** 每条核心决策应以稳定编号前缀开头，格式为 `> **[REQ-{topic}-NNN]**`（如 `> **[REQ-权限-001]** 系统采用 RBAC 权限模型`），以便下游技能引用追溯。序号从 001 开始递增，同一结论不换号，新结论用新号。

## 二、详细设计

### 2.1 {子主题}

[具体设计内容，仅保留最新结论]

### 2.2 {子主题}

...

## 三、仍待决策

[如果有 ❓待定 或 TODO 项，集中列在这里。没有则省略此节]

## 四、来源索引

| 原始文件 | 主要贡献内容 |
|---------|-------------|
| {source-file-1}.md | {该文件对本文档的主要贡献} |
| {source-file-2}.md | {同上} |
```

#### 6.2 写作原则

| 原则 | 说明 |
|------|------|
| **仅保留最新结论** | 不写"最初设计是 A，后来改为 B"——直接写 B |
| **不解释为什么改了** | 除非该理由本身就是重要的设计约束 |
| **使用肯定语气** | "X 是 Y"，而非"X 应该是 Y"或"我们觉得 X 是 Y" |
| **删除过程标记** | 已确认的内容不需要 ✅/❓ 标记 |
| **标注不确定性** | 仍未决定的项，用 `⚠️ 待定：` 前缀明确标注，不与已确认内容混淆 |
| **精确引用** | 枚举值、字段名、类型名必须与原始文件中的表述一致 |
| **不推测** | 原始文件中没有明确确认的内容，不自行推断为"最新结论" |
| **合并重复** | 多个文件重复确认同一结论 → 只写一次，用最精确的表述 |
| **语言一致** | 整合文档使用与原始输入相同的语言（中文输入→中文输出；英文输入→英文输出） |
| **为下游追溯提供锚点** | 每条独立的设计结论应有稳定编号（如 `[REQ-{topic}-NNN]`），下游技能通过此编号追溯。编号规则：`REQ-{topic-slug}-{序号}`。同一结论不换号，新结论用新号 |

#### 6.3 文件命名规范

- 使用两位数前缀 + 主题短横线命名：`{NN}-{topic-slug}.md`
- 前缀数字表示建议的阅读顺序，无其他含义
- 全局/跨领域内容放在 `00-global-design-decisions.md`

### 步骤 7：清理上一轮日志（建议）

在将新一批原始文件移入 `original-logs/` 之前，**建议**先清理上一轮已归档的旧文件。

**默认行为：** 在步骤 5 确认方案时，主动向用户提出：

> "在移入新的原始文件之前，建议先清空 `original-logs/` 中上一轮归档的旧文件。
> 这样可以保持 `original-logs/` 精简——它始终只保留最近一轮整理前的原始快照。
> 更早的归档文件在 Git 历史中可随时恢复。是否清理？"

**用户确认后执行：**

1. **Git 安全检查（必须）：** 删除前先确认 `original-logs/` 中的所有文件已提交到代码库：
   ```bash
   git status --short {raw-input-dir}/original-logs/
   ```
   - 如果输出为空或仅有 `_archive-manifest.md` 的修改 → 安全，继续
   - 如果存在**未跟踪（`??`）或未提交（` M`/`A `/` D`）的文件** → 停止并警告用户：
     > "⚠️ `original-logs/` 中有尚未提交到 Git 的文件：{列出文件}。
     > 如果现在删除，这些文件将永久丢失（无法从 Git 恢复）。
     > 建议先 `git commit` 这些文件，或将其移到其他位置暂存。是否仍然删除？"
   - 用户确认"仍然删除"后才可继续
2. 删除 `original-logs/` 中除 `_archive-manifest.md` 以外的所有文件
3. 保留 `_archive-manifest.md`（追加新批次信息，形成归档历史链）

**用户拒绝时：** 跳过清理，新文件直接追加到 `original-logs/`（新旧文件共存）。

**注意：** 这是默认建议，不是强制行为。如果用户明确说"先不清理旧日志"，遵从用户意愿。

### 步骤 8：归档（Archive）

写入完成后，执行归档：

1. 创建 `{raw-input-dir}/original-logs/` 目录（如不存在）
2. 将方案中标记为"归档"的原始文件**移动**（不是复制）到 `original-logs/`
3. 在 `original-logs/` 下创建或更新 `_archive-manifest.md`：

   ```markdown
   # 归档清单
   > 归档日期：YYYY-MM-DD
   > 归档批次：{简要描述本次整理了哪些内容}

   | 原始文件 | 归档日期 | 内容并入 |
   |---------|---------|---------|
   | {file-a}.md | YYYY-MM-DD | {topic-slug-1}.md, {topic-slug-2}.md |
   | {file-b}.md | YYYY-MM-DD | {topic-slug-1}.md |
   | ... | ... | ... |
   ```

4. **确认**：归档后 `{raw-input-dir}` 根目录下只剩下整合文档 + 方法论文件 + `original-logs/` 子目录

## 归档后的目录结构（示意）

```
{raw-input-dir}/
├── 00-global-design-decisions.md   ← 整合：跨领域的全局决策
├── {NN}-{topic-a}.md               ← 整合：某业务主题
├── {NN}-{topic-b}.md               ← 整合：某业务主题
├── {NN}-{topic-c}.md               ← 整合：某业务主题
├── {methodology-file}.md           ← 保留：方法论参考
├── original-logs/                  ← 归档目录
│   ├── _archive-manifest.md
│   ├── {original-discussion-1}.md
│   ├── {original-discussion-2}.md
│   ├── {audio-file}.wav
│   └── ...
```

> 实际整合文档的数量、命名和分组由步骤 4 的分析结果决定。
> 不同项目的领域划分不同，整合文档的结构应反映项目自身的业务主题，而非套用固定模板。

## 常见场景处理

### 场景 A：用户只整理部分主题

如果用户指定了范围（如"只整理权限相关的原始输入"）：

1. 仅扫描和分类涉及指定主题的文件
2. 其他文件保持不动
3. 整合方案中明确标注处理范围

### 场景 B：已存在整合文档

如果 `{raw-input-dir}` 中已有之前生成的整合文档：

1. 将已有整合文档也纳入分析范围（它们可能已被新的原始讨论覆盖）
2. 已有整合文档的内容被新讨论更新 → 更新整合文档
3. 已有整合文档仍然准确 → 保留，只合并新增的原始文件

### 场景 C：方法论文件被误归档

如果用户反馈某个文件不应该被归档：

1. 从 `original-logs/` 移回根目录
2. 更新 `_archive-manifest.md`

### 场景 D：需要恢复某个原始文件

- 原始文件在 `original-logs/` 中，可直接移回
- Git 历史中有完整备份，可随时恢复
- 用户可定期清理 `original-logs/` 中的旧文件

### 场景 E：清理旧日志时用户想保留某些文件

如果用户希望保留 `original-logs/` 中的部分旧文件（比如某次重要决策的原始记录）：

1. 在清理前询问哪些文件需要保留
2. 将需保留的文件移回根目录（作为参考）或单独标注
3. 其余旧文件正常清理

## 与其他技能的关系

本技能整理的是**原始设计讨论记录**。在一个典型的设计资产驱动项目中：

- **设计类技能**（工作流设计、ER 设计、API 设计等）——在流程中向原始输入目录**追加**新的讨论记录
- **复查/验证类技能**——复查讨论也存档于原始输入目录
- **本技能（整理）**——定期清理上述技能积累的原始记录

上述技能是自动化的流水线行为，本技能是**手动触发的清理行为**。两者独立运作，互不依赖。

## 约束与红线

| # | 规则 |
|---|------|
| 1 | **不自动执行** — 必须用户明确调用（触发词匹配）才启动 |
| 2 | **先规划后执行** — 步骤 4 的整合方案必须经用户确认后才能写入和归档 |
| 3 | **不丢失信息** — 删除日志前必须通过 `git status` 确认已提交；未提交的文件不允许删除。Git 历史是唯一永久备份 |
| 4 | **不修改原始文件内容** — 归档是移动操作，不编辑原始文件 |
| 5 | **方法论文件不归档** — 定义工作方法的文档保留在原位置 |
| 6 | **不推测结论** — 原始文件中没有明确确认的内容，在整合文档中标注为"待定" |
| 7 | **整合文档可迭代** — 后续新的原始讨论可以触发再次整理，更新已有整合文档 |
| 8 | **不预设领域结构** — 整合文档的拆分方式由分析结果决定，不套用项目领域注册表或其他既有分类 |
| 9 | **默认清理旧日志** — 每次归档前建议清空上一轮旧文件（前提：通过 Git 安全检查） |
| 10 | **不破坏追溯锚点** — 如果已有整合文档中含 `[REQ-*]` 编号，更新时必须保留编号稳定性 |
